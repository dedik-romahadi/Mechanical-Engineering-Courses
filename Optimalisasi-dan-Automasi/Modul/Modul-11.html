Bagian 01

Mengapa Rule-Based:
Monitoring Real-Time yang Sederhana, Transparan & Reliable

Setelah mesin dioptimasi (P9–P11) dan dipasang di lini produksi, pertanyaan beralih dari "parameter mana yang terbaik?" ke "apakah mesin masih sehat saat ini?". Inilah problem kondisi mesin (machine health): detect anomali sebelum failure, kirim alert ke operator, log riwayat untuk trend analysis. Ada banyak pendekatan: (1) rule-based (threshold sederhana, IF-THEN), (2) machine learning (anomaly detection, classifier), (3) deep learning (autoencoder, LSTM forecasting), (4) fisika-based (digital twin). Yang paling banyak dipakai di industri: rule-based — tidak glamour, tetapi reliable, transparan, dan mudah audit. ISO 10816 (vibrasi), ISO 13373 (vibration condition monitoring), API 670 (turbomachinery) — semua menetapkan threshold rule-based. Dipakai oleh ABB, Siemens, GE, Honeywell sebagai baseline sebelum upgrade ke ML/DL. Pelajari rule-based dengan benar = bekal kuat untuk seluruh karier QC/maintenance engineering.

Lima Karakteristik Rule-Based Alerting
Transparan (auditable)  ·  Deterministik (reproducible)  ·  Real-time (latency < 1 detik)  ·  Compliance-ready (ISO/API)  ·  Compute murah (microcontroller-friendly)
🔍

Transparan & Auditable

Setiap keputusan alert bisa dijelaskan: "karena rms = 8.2 mm/s > threshold 7.1 mm/s, status = DANGER". Auditor ISO/API hanya butuh review tabel rule. Bandingkan ML black-box (random forest, neural network) yang butuh SHAP/LIME untuk explainability. Untuk industri regulated (nuclear, aviation, medical), rule-based wajib karena root-cause traceability 100% tanpa ambiguitas.

IF condition THEN action
🎯

Deterministik & Reproducible

Untuk input yang sama, output selalu sama — tidak ada randomness. Berbeda dari ML yang punya noise pelatihan + inference. Konsekuensi positif: bug mudah direproduksi (input X selalu trigger alert; jika tidak, ada bug). Konsekuensi negatif: tidak adaptif terhadap drift (perlu re-tune threshold manual). Modern best practice: rule-based core + ML adaptive layer (Pertemuan 13).

f(x_t) = const → repeatable
⚡

Real-Time & Edge-Friendly

Threshold check = 1 instruksi CPU (nanodetik). Bisa berjalan di edge device: PLC, Raspberry Pi, ESP32, microcontroller 8-bit. Tidak butuh GPU, internet, atau cloud. Latency end-to-end (sensor → alert) < 100 ms. Bandingkan ML inference yang biasanya butuh 10–100 ms hanya untuk forward pass + butuh memory lebih besar. Untuk safety-critical (turbine over-speed), rule-based di hardware-level (ASIC) wajib.

latency < 100 ms (edge)
📜

Compliance-Ready (ISO/API)

Standar industri sudah menetapkan threshold:
• ISO 10816-3: vibrasi 4 zona (A: good ≤ 1.4 mm/s, B: acceptable, C: unsatisfactory, D: danger ≥ 7.1 mm/s) untuk mesin 15–300 kW.
• API 670: turbomachinery vibrasi alert 6.4 mm/s, danger 12.7 mm/s.
• ISO 13373-3: bearing severity (kurtosis, crest factor) thresholds. Compliance audit hanya butuh tunjukkan implementation match standard.

Bagian 02

Threshold & ISO 10816 Zone Classification:
Standar Industri untuk Vibrasi Mesin

Threshold adalah nilai batas yang memisahkan "OK" dari "anomali". Pertanyaan kunci: berapa nilainya? Jangan tebak. Ada 3 sumber threshold: (1) Standar internasional — ISO 10816 (vibrasi), API 670 (turbomachinery), AGMA (gear), ANSI/HI (pump). (2) Datasheet vendor — bearing manufacturer (SKF, FAG) menetapkan limits. (3) Empirical baseline — ukur mesin sehat selama 30 hari, ambil μ + 3σ. Yang paling banyak dipakai untuk vibrasi mesin rotasi: ISO 10816-3 (rotating machines 15 kW – 50 MW). Memetakan amplitude RMS velocity (mm/s) ke 4 zona (A: good, B: acceptable, C: unsatisfactory, D: danger) berdasarkan kelas mesin (small/medium/large) & foundation (rigid/flexible). Implementasi rule-based ISO 10816 = standar minimum untuk monitoring.

Enam Konsep Inti Threshold & Zone Classification
Threshold absolut  ·  ISO 10816-3 Zona A/B/C/D  ·  Class I–IV (mesin)  ·  Rigid vs Flexible foundation  ·  RMS velocity 10 Hz – 1 kHz  ·  Multi-feature threshold (RMS + peak + kurtosis)
📐

Threshold Absolut: T_low & T_high

Bentuk paling sederhana: IF rms > T_high THEN ALERT. Untuk fitur dua-arah (mis. tekanan oli yang tidak boleh terlalu rendah dan terlalu tinggi): IF p < T_low OR p > T_high THEN ALERT. Threshold harus physical-meaningful: didasarkan pada batas mekanis (yield stress, bearing fatigue limit), standar industri, atau baseline operasi normal. Hindari threshold ditebak — selalu dokumentasikan sumbernya.

if x > T_high: alert
🏛️

ISO 10816-3 — Zone A/B/C/D

Standar internasional untuk vibrasi mesin rotasi. Pakai RMS velocity (mm/s) di band 10 Hz–1 kHz. Untuk Group 2 (medium 15–300 kW, rigid foundation): A (Good) ≤ 1.4, B (Acceptable) 1.4–2.8, C (Unsatisfactory) 2.8–4.5, D (Danger) > 4.5. Mesin di zona D harus shutdown atau diperbaiki segera. Klasifikasi otomatis = lookup table sederhana di Python: if rms ≤ 1.4: 'A' elif rms ≤ 2.8: 'B' elif rms ≤ 4.5: 'C' else: 'D'.

A | B | C | D zones
🔧

Class I–IV (Group Mesin)

Threshold ISO 10816-3 berbeda per kelas: Class I (small < 15 kW: motor pump kecil), Class II (medium 15–75 kW: blower, kompresor kecil), Class III (large machines on rigid foundations: turbin uap), Class IV (large machines on flexible foundations: paper machine, large compressor). Mesin yang sama bisa "OK" di Class IV (toleransi besar) tetapi "DANGER" di Class I (toleransi ketat). Selalu cek kelas mesin sebelum pilih threshold.

Class I (small) → IV (large flex)
🏗️

Rigid vs Flexible Foundation

Foundation = pondasi mesin. Rigid (concrete masif, baseplate steel tebal): vibrasi yang ditransmit kecil → threshold lebih ketat. Flexible (skid mounting, isolator karet, struktural baja): foundation menyerap vibrasi → threshold lebih longgar. ISO 10816 punya tabel terpisah untuk keduanya. Rule of thumb: kalau natural frequency foundation > 25% frequency mesin, considered rigid; kalau di bawah, flexible.

f_n_found vs f_machine
📡

RMS Velocity 10 Hz – 1 kHz

ISO 10816 spesifik mengukur RMS velocity (bukan acceleration atau displacement) di band 10 Hz – 1 kHz. Pilihan ini berdasarkan: (1) velocity berkorelasi dengan energi kinetik vibrasi (≈ kerusakan), (2) band 10 Hz–1 kHz cover frekuensi mekanis dominan (running speed, gear meshing, bearing characteristic). Untuk mesin di luar range ini (mis. high-speed turbomachinery di 5–10 kHz), pakai standar lain (API 670: acceleration band > 5 kHz).

RMS_v = √(mean(v²)) di 10 Hz–1 kHz
🔬

Multi-Feature Threshold (RMS + Peak + Kurtosis)

Threshold tunggal (RMS) bisa miss anomali tertentu. Kombinasi multi-feature lebih sensitif: RMS deteksi unbalance/misalignment; peak amplitude deteksi impact (bearing pitting); kurtosis > 4 deteksi distribusi non-Gaussian (early bearing fault). Rule kombinasi: IF rms > T1 OR peak > T2 OR kurt > T3 THEN ALERT. Tools: P1–P7 (extract fitur) lalu P12 (apply rule). Kompleksitas grow → next step: ML (P13).

Bagian 03

Hysteresis (Anti-Chattering):
Schmitt Trigger untuk Threshold yang Stable

Threshold tunggal (x > T → ALERT) punya masalah serius di sistem real: chattering. Saat sinyal sensor di sekitar threshold (T ± noise level), state berkedip cepat ON/OFF/ON/OFF — operator dibanjiri alert palsu, log file penuh dengan transition spam. Solusi klasik dari electronics: hysteresis (dua threshold T_on > T_off). Untuk turn ON butuh signal > T_on; untuk turn OFF butuh signal < T_off. Margin (T_on − T_off) ≥ 2× noise level → chattering hilang. Konsep dari Schmitt trigger (Otto Schmitt 1934) yang dipakai di setiap power MOSFET driver, thermostat, comparator IC. Untuk tier alarm escalation (warning < alert < critical) dan finite state machine, lihat modul berikutnya (Pertemuan 12 — Monitoring Part 2).

Lima Konsep Inti Hysteresis & Tier Alarm
Hysteresis (T_on > T_off)  ·  Schmitt trigger analogy  ·  Tier alarm (3 level)  ·  Escalation timer  ·  Alarm flood prevention (ANSI/ISA 18.2)
↔️

Hysteresis: T_on > T_off

Definisi: dua threshold berbeda untuk transition ke "ON" vs ke "OFF". Saat status OFF, perlu sinyal > T_on untuk pindah ke ON. Saat status ON, sinyal harus turun di bawah T_off untuk kembali ke OFF. Tipikal: T_off = T_on − margin (mis. T_on=70, T_off=65). Margin 5–15% biasanya cukup. Tanpa hysteresis, sinyal di sekitar threshold ber-flip-flop sangat cepat (chattering).

T_on = 70, T_off = 65 (margin 5)
⚡

Schmitt Trigger Analogy

Konsep hysteresis dipakai di rangkaian elektronik Schmitt trigger (Otto Schmitt 1934) — comparator dengan dua threshold. Awalnya untuk filter noise di radar WWII, sekarang dipakai di setiap microcontroller (Arduino, STM32) untuk debouncing button input. Algoritma yang sama persis bisa diterapkan untuk threshold-based monitoring di software. Insight: setiap masalah noise-induced chattering punya solusi hysteresis yang elegan.

analog Schmitt trigger
🎚️

Tier Alarm — Tiga Level Eskalasi

Bukan binary OK/FAIL, tetapi gradient 3+ level: WARNING (yellow, perhatikan) → ALERT (orange, action segera) → CRITICAL (red, shutdown). Tiap tier punya threshold sendiri: T_warn < T_alert < T_critical. Aksi berbeda: warning → log + notification email; alert → SMS ke supervisor; critical → auto-shutdown + page on-call engineer. Memetakan ke ISO 10816: A (no alarm), B (warning), C (alert), D (critical).

T_warn < T_alert < T_crit
⏱️

Escalation Timer (Persistence)

Beberapa anomali muncul sesaat (transient noise) dan menghilang sendiri — tidak perlu eskalasi. Tetapi kalau anomali persist > T detik, tier perlu naik. Contoh: warning persist 5 menit → eskalasi ke alert; alert persist 10 menit → eskalasi ke critical. Timer reset ketika sinyal kembali normal (atau hysteresis turun). Implementasi: state machine dengan timer per state.

if t_in_state > Δt: escalate
🔇

Alarm Flood Prevention (ANSI/ISA 18.2)

Bahaya nyata: alarm flood — saat 1 root cause memicu cascade puluhan alarm sekaligus, operator overwhelmed. ANSI/ISA 18.2 (Alarm Management) menetapkan limit: ≤ 1 alarm/10 menit per operator (steady state); ≤ 10 alarm dalam 10 menit pertama post-disturbance. Solusi: alarm grouping (suppress child alarm jika parent sudah on), shelving (operator bisa silence alarm selama window terbatas), alarm rationalization (review periodik untuk delete alarm yang tidak actionable).

≤ 1 alarm / 10 min
🎨

HMI Color Convention

Standar industri (ABB 800xA, Siemens PCS 7, Honeywell Experion) untuk warna alarm: hijau (running normal), cyan/biru (info / state change), kuning (warning), orange/oranye (alert), merah (critical), ungu (failure / no signal). Konsistensi warna penting untuk operator yang shift-rotate antar mesin/pabrik. Gunakan ikon (✓ ⚠ ✗) di samping warna untuk accessibility (color-blind operators).

💡 Implementasi Hysteresis di Python (state machine sederhana):
class HysteresisAlarm:
    def __init__(self, t_on, t_off):
        self.t_on, self.t_off = t_on, t_off
        self.state = False
    def update(self, x):
        if not self.state and x > self.t_on: self.state = True
        elif self.state and x < self.t_off: self.state = False
        return self.state
Untuk tier alarm, extend ke 3 instance (warn/alert/crit) atau 1 class dengan list threshold. Frameworks production: OpenSCADA, Ignition, WinCC.
Bagian 04

Sensor Selection & Akuisisi Data:
Hardware Foundation Monitoring System

Sebelum threshold detection bisa bekerja, harus ada data sensor berkualitas yang masuk ke sistem. Pilihan sensor yang salah, sampling rate kurang, atau wiring yang noisy = garbage in, garbage out. Pertemuan ini bahas fondasi hardware monitoring: jenis sensor (accelerometer piezoelectric/MEMS untuk vibrasi, RTD/thermocouple untuk suhu, current transformer untuk motor, flow/pressure transmitter), sampling rate (Nyquist criterion: f_s ≥ 2 × f_max signal), DAQ hardware (NI USB-6009 industrial-grade, Arduino + MPU6050 entry-level lab, Raspberry Pi + ADS1115 mid-range), signal conditioning (charge amplifier, anti-aliasing filter, ADC resolution 12–24 bit), dan protokol komunikasi (Modbus RTU/TCP, OPC UA, MQTT, RS-485). Pemilihan harus match dengan karakteristik mesin (frekuensi vibrasi dominan), environment (suhu, kelembaban, EMI), dan budget.

Enam Komponen Akuisisi Data Industrial
Sensor (transducer)  ·  Signal conditioning  ·  ADC resolution  ·  Sampling rate (Nyquist)  ·  Protokol komunikasi  ·  Edge gateway / DAQ
📳

Accelerometer Vibrasi

Sensor utama untuk vibrasi mesin. Dua jenis: Piezoelectric (IEPE) — high frequency response (1 Hz – 30 kHz), durable, butuh constant current excitation 4 mA. Contoh: PCB Piezotronics 352C33 (~Rp 5–15 juta). Pakai untuk industrial heavy machinery. MEMS — low frequency (DC – 1 kHz), murah, integrated chip. Contoh: ADXL355 (Rp 200K), MPU6050 (Rp 30K). Pakai untuk lab eksperimen, low-cost monitoring. Mounting wajib rigid (stud mount > magnetic mount > adhesive) untuk akurasi high frequency. Pasang di bearing housing, arah radial + axial.

IEPE 4 mA · MEMS digital I2C/SPI
🌡️

Temperature: RTD & Thermocouple

RTD (Pt100/Pt1000) — resistance varies dengan suhu, accuracy ±0.1°C, range −200 to +850°C, response slow (seconds). Pakai untuk bearing temperature, oil temperature. Wiring 3-wire atau 4-wire untuk cancel lead resistance. Thermocouple (Type K, J, T) — voltage millivolt dari junction dissimilar metals, range −270 to +1372°C (Type K), response fast (millisecond), accuracy ±1°C. Pakai untuk hot gas, exhaust, kiln temperature. Cold junction compensation wajib. Output mV langsung butuh amplifier (gain ~100×).

RTD: R(T); TC: V(T)
⚡

Current Transformer (CT) Motor

Untuk monitor health motor, ukur arus stator. Sensor: Current Transformer (split-core CT, clamp-on, no wiring break). Output current AC proporsional dengan primary current. Ratio tipikal 100:5, 1000:5. Pakai dengan burden resistor untuk convert ke voltage. Modern: Hall-effect sensor (LEM, Allegro) untuk DC + AC, no contact dengan conductor. Aplikasi: Motor Current Signature Analysis (MCSA) — broken rotor bar, eccentricity, bearing fault detect dari spectrum arus. Threshold: arus naik > 10% baseline = warning.

I_secondary = I_primary / N
📊

Sampling Rate: Nyquist Criterion

Aturan emas: f_sampling ≥ 2 × f_max (Shannon-Nyquist 1949). Untuk capture vibrasi sampai 1 kHz, butuh f_s ≥ 2 kHz (industri pakai f_s = 5 kHz untuk margin). Aliasing: kalau signal di atas Nyquist, frekuensi tinggi "lipat balik" jadi frekuensi rendah palsu — interpretasi salah. Solusi: anti-aliasing filter (low-pass analog, cutoff f_max) sebelum ADC. Best practice: f_s = 2.5–5 × f_max signal. Untuk ISO 10816 (10 Hz – 1 kHz), f_s = 2500–5000 Hz cukup.

f_s ≥ 2 × f_max
🔌

Tier 2 — Industrial: Bently Nevada 3500 series, Emerson AMS Machine Works (vibration monitoring rack-mount). Untuk capstone project mahasiswa, Tier 3 sangat cukup. Production deployment butuh Tier 1/2.

budget vs accuracy trade-off

Protokol Komunikasi Industrial

DAQ → server / SCADA pakai protokol standar: Modbus RTU (RS-485, simple, dipakai 1979–sekarang) untuk PLC. Modbus TCP (Ethernet variant). OPC UA (modern industry 4.0 standard, secure, semantic data) — jadi default ABB, Siemens. MQTT (publisher/subscriber, IoT lightweight) untuk wireless sensor + cloud. HART (4–20 mA + digital overlay) untuk legacy sensor pressure/flow. Pemilihan: legacy plant → Modbus, new plant → OPC UA, IoT/wireless → MQTT. Latency: Modbus ~10–100 ms, MQTT ~50–500 ms (cloud roundtrip).

Analogi
Rule-Based Monitoring di Sehari-hari

Threshold + hysteresis + state machine bukan konsep eksotik — kita berinteraksi dengan rule-based system setiap hari. Termostat AC, indikator bensin mobil, alarm asap, smartwatch alarm detak jantung — semua adalah rule-based monitoring dengan threshold dan kadang hysteresis. Enam analogi berikut menunjukkan bagaimana rule-based logic muncul di skenario sehari-hari, dari yang paling sederhana sampai industrial-grade.

🌡️ Analogi 1 — Termostat AC (Hysteresis Klasik) AC di rumah Anda set ke 24°C. Kalau threshold tunggal: AC nyala saat suhu = 24.01°C, mati saat 23.99°C → flicker on/off setiap detik (chattering, kompressor cepat rusak). Solusi: hysteresis 1°C — AC nyala saat suhu > 25°C, mati saat suhu < 23°C. Margin 2°C jauh > noise sensor. Hasil: AC siklus tiap 5–10 menit, awet, suhu rata-rata stabil di 24°C. Termostat = Schmitt trigger fisik sejak 1885 (Albert Butz, Honeywell pendiri). Engineering wisdom yang sama persis dipakai di industrial monitoring.
⛽ Analogi 2 — Indikator Bensin Mobil (Tier Alarm) Dashboard mobil punya 3 level alarm: jarum di 1/4 tangki (warning ringan, sudah waktunya cari SPBU), jarum di garis E + icon pump kuning (alert serius, ~10 km lagi), buzzer + lampu merah (critical, mungkin sudah cuma 5 km). Persis tier alarm: warning → alert → critical, eskalasi gradual sesuai tingkat keparahan. Plus debouncing: kalau jalan goyang membuat sensor flicker, dashboard tidak panik. Plus heartbeat: kalau sensor mati total, "---" muncul. Mobil modern (Tesla, BMW iDrive) implement FSM lengkap untuk tank monitoring.
🚨 Analogi 3 — Alarm Asap di Rumah (Debouncing) Smoke detector standar UL 217 / EN 14604 didesain untuk tidak alarm dari uap mandi atau asap rokok sesaat. Bagaimana? Debouncing: alarm hanya trigger jika konsentrasi smoke particle > threshold selama ≥ 30 detik kontinyu (N consecutive samples). Single peak (mis. orang lewat dengan kompor portabel sebentar) tidak trigger. Tetapi kalau persist (kebakaran genuine), buzzer 85 dB nyala. Plus low-battery beep tiap 30 detik (heartbeat / sensor health). Ini rule-based system yang menyelamatkan ribuan nyawa per tahun.
🔋 Analogi 4 — Smartwatch Heart Rate Alarm (Multi-Threshold) Apple Watch dan Fitbit memantau detak jantung 24/7. Alarm rule: HR > 120 bpm saat tidak olahraga selama 10 menit (high), HR < 40 bpm selama 10 menit (low). Tier: notification (warning) → call emergency contact (alert) → 911 (critical). Persis FSM medis: NORMAL → ABNORMAL → AFib detected → emergency. Multi-feature: HR + ECG + activity context. FDA approval (Apple Watch ECG 2018) wajib FSM yang validated dengan low false-positive rate (Apple claim < 0.5%). Saved-life stories sering muncul di media setiap tahun.
🚦 Analogi 5 — Lampu Lalu Lintas (FSM Klasik) Lampu lalu lintas adalah finite state machine sempurna: 3 state (Hijau, Kuning, Merah) dengan transisi deterministik (Hijau → Kuning setelah 30s; Kuning → Merah setelah 5s; Merah → Hijau setelah 30s). Pintar: cross-direction interlock (Hijau utara-selatan IMPLIES Merah timur-barat). Adaptif modern: FSM + sensor (induksi loop deteksi mobil) + timer adaptif (lebih banyak mobil → fase Hijau lebih panjang). FSM yang sama logic di-embedded ke Arduino sederhana. Pelajari FSM dengan benar, Anda bisa bangun apa saja dari traffic light sampai mesin industri.
📊 Analogi 6 — DevOps Server Monitoring (Production-Grade) Tim DevOps Netflix, Google, AWS memantau jutaan server real-time. Rule-based system: CPU > 80% selama 5 menit → warning (PagerDuty Slack); CPU > 95% selama 2 menit → alert (page on-call engineer); response time > 5s → critical (auto-failover). Hysteresis 5–10% untuk hindari flapping alert. Tools: Prometheus + Alertmanager, Grafana, Datadog, New Relic. Sama persis dengan industrial monitoring di pabrik — hanya beda domain (CPU/latency vs vibrasi/temperatur). Lulusan Teknik Mesin yang paham rule-based monitoring sangat valuable di SRE/DevOps roles juga.
💡 Pola yang Sama di Semua Analogi: Rule-based monitoring bekerja karena kombinasi 3 elemen: (1) threshold dengan referensi jelas (suhu setpoint, fuel level, smoke ppm, HR bpm, CPU%, RMS vibrasi); (2) noise filtering (hysteresis untuk thermostats, debouncing untuk smoke alarm, minimum duration untuk medical alarm); (3) graceful escalation (multi-tier dari informasi → urgent → critical). Insight bagi engineer: setiap monitoring system yang sukses pasti punya 3 elemen ini. Kalau Anda merancang monitoring tanpa hysteresis, tanpa debouncing, tanpa tier — siap-siap operator complaint dengan alarm flood. Tetapi rule-based punya batas: jika ada 50 fitur dan 100 mode kegagalan, manual rule tidak scalable. Saatnya pindah ke ML (Pertemuan 13) — tetapi baseline rule-based selalu layak dipertahankan sebagai safety net.

Jebakan Praktis Rule-Based Monitoring di Industri

(1) Threshold ditebak tanpa referensi — selalu cite ISO/API/datasheet. Kalau pakai empirical baseline (μ + 3σ), dokumentasikan periode & sample size. (2) Lupa hysteresis — hampir setiap newcomer engineer membangun rule tanpa hysteresis, lalu komplain "alert spam". Hysteresis = mandatory. (3) Single feature monitoring — RMS saja sering miss bearing fault (kurtosis lebih sensitif). Pakai multi-feature minimum 2 (RMS + peak atau RMS + kurtosis). (4) Tidak ada sensor health check — heartbeat & range validation (sensor reading di luar physically possible range = sensor failure, bukan reading valid). (5) Alarm flood ignored — ANSI/ISA 18.2 limit ≤ 1 alarm/10 min/operator. Lebih dari itu, operator alarm fatigue → ignore semua alarm. (6) Tidak ada alarm rationalization — review periodik (6–12 bulan): alarm yang tidak pernah actionable harus dihapus. (7) Threshold tidak adaptif drift — komponen mesin baru beda dari aged. Re-tune threshold setelah 6 bulan operational atau setelah major maintenance. Modern best practice: rule-based core + adaptive threshold via ML (Pertemuan 13).

Cell 2 — HysteresisAlarm Class: Hindari Chattering dengan Dual Threshold (T_on / T_off) Python 
np.random.seed(42)

# ═══════════════════════════════════════════════════════════════════
# SCENARIO: Sinyal noisy di sekitar threshold (boundary case)
#   Tanpa hysteresis: chattering (alarm flicker)
#   Dengan hysteresis: stable transition
#   Demo: T_on = 2.8 (Zone B→C), T_off = 2.6 (margin 0.2)
# ═══════════════════════════════════════════════════════════════════

class HysteresisAlarm:
    """Threshold dengan hysteresis untuk hindari chattering."""
    def __init__(self, t_on, t_off, name='alarm'):
        assert t_on > t_off, "T_on harus > T_off untuk hysteresis"
        self.t_on = t_on
        self.t_off = t_off
        self.name = name
        self.state = False
        self.transitions = 0

    def update(self, x):
        if not self.state and x > self.t_on:
            self.state = True
            self.transitions += 1
        elif self.state and x < self.t_off:
            self.state = False
            self.transitions += 1
        return self.state

# Simple threshold (no hysteresis) untuk komparasi
class SimpleThreshold:
    def __init__(self, t):
        self.t = t
        self.state = False
        self.transitions = 0
    def update(self, x):
        new_state = x > self.t
        if new_state != self.state:
            self.transitions += 1
        self.state = new_state
        return self.state

# Generate sinyal di sekitar boundary (rms ≈ 2.7 ± 0.15 noise)
n = 500
t = np.arange(n)
rms = 2.7 + 0.15 * np.random.randn(n)        # Sinyal di sekitar threshold
rms[200:350] += 0.5                              # Genuine elevation di tengah

# Apply both alarms
hyst = HysteresisAlarm(t_on=2.8, t_off=2.6, name='WarningC')
simple = SimpleThreshold(t=2.8)

state_hyst = []
state_simple = []
for r in rms:
    state_hyst.append(hyst.update(r))
    state_simple.append(simple.update(r))

state_hyst = np.array(state_hyst, dtype=int)
state_simple = np.array(state_simple, dtype=int)

print("═══ COMPARISON: Simple Threshold vs Hysteresis ═══")
print(f"  Simple threshold transitions: {simple.transitions}  (chattering!)")
print(f"  Hysteresis transitions       : {hyst.transitions}  (stable)")
print(f"  Reduksi spam: {100*(1 - hyst.transitions/simple.transitions):.0f}%")

#   FSM dengan 5 state: NORMAL → WARNING → ALERT → CRITICAL + FAULT
#   Debouncing: butuh N=3 consecutive samples untuk transition
#   Tier thresholds: T_warn=1.4, T_alert=2.8, T_crit=4.5 (ISO 10816)
#   Hysteresis: T_off = T_on - 0.2
#   Audit log: timestamp, old_state, new_state, trigger_value
# ═══════════════════════════════════════════════════════════════════

class MonitoringFSM:
    """Finite State Machine untuk monitoring tier alarm dengan debouncing."""

    STATES = ['NORMAL', 'WARNING', 'ALERT', 'CRITICAL', 'FAULT']
    THRESHOLDS = {'WARNING': 1.4, 'ALERT': 2.8, 'CRITICAL': 4.5}
    HYSTERESIS = 0.2
    DEBOUNCE_N = 3

    def __init__(self):
        self.state = 'NORMAL'
        self.up_counter = 0           # consecutive samples above threshold
        self.down_counter = 0         # consecutive samples below threshold
        self.audit_log = []
        self.last_sample_time = None

    def _target_state(self, x):
        """Tentukan state target berdasarkan threshold (tanpa hysteresis)."""
        if x >= self.THRESHOLDS['CRITICAL']:
            return 'CRITICAL'
        elif x >= self.THRESHOLDS['ALERT']:
            return 'ALERT'
        elif x >= self.THRESHOLDS['WARNING']:
            return 'WARNING'
        else:
            return 'NORMAL'

    def update(self, x, ts=None):
        """Process satu sample, return current state."""
        if ts is None:
            ts = datetime.now()
        self.last_sample_time = ts

        # Sensor health: kalau reading tidak masuk akal → FAULT
        if x < 0 or x > 50:
            self._transition('FAULT', x, ts, reason='Out-of-range sensor')
            return self.state

        target = self._target_state(x)
        target_idx = self.STATES.index(target)
        current_idx = self.STATES.index(self.state) if self.state != 'FAULT' else 0

        # Debouncing: increment counter only if direction consistent
        if target_idx > current_idx:
            self.up_counter += 1
            self.down_counter = 0
            if self.up_counter >= self.DEBOUNCE_N:
                self._transition(target, x, ts, reason='Escalation')
                self.up_counter = 0
        elif target_idx < current_idx:
            # Apply hysteresis: turun state hanya kalau x < T - hysteresis
            below_off = self._below_off_threshold(x, current_idx)
            if below_off:
                self.down_counter += 1
                self.up_counter = 0
                if self.down_counter >= self.DEBOUNCE_N:
                    self._transition(target, x, ts, reason='Recovery')
                    self.down_counter = 0
        else:
            self.up_counter = 0
            self.down_counter = 0

        return self.state

    def _below_off_threshold(self, x, current_idx):
        """Cek apakah x di bawah T_off (T - hysteresis) dari state saat ini."""
        if current_idx == 0:
            return False          # Already at NORMAL
        current_name = self.STATES[current_idx]
        t_off = self.THRESHOLDS[current_name] - self.HYSTERESIS
        return x < t_off

    def _transition(self, new_state, trigger, ts, reason=''):
        """Eksekusi state transition + log."""
        old_state = self.state
        self.state = new_state
        self.audit_log.append({
            'timestamp': ts,
            'old_state': old_state,
            'new_state': new_state,
            'trigger_value': round(trigger, 3),
            'reason': reason,
        })

Cell 4 — run_monitoring_pipeline() Reusable: End-to-End Pipeline (FSM + dashboard) Python 
def run_monitoring_pipeline(rms_signal, timestamps=None,
                            thresholds=(1.4, 2.8, 4.5),
                            hysteresis=0.2, debounce_n=3,
                            plot=True):
    """
    End-to-end rule-based monitoring pipeline.
    
    Steps:
      1. Apply ISO 10816 zone classification
      2. Apply hysteresis-aware FSM with debouncing
      3. Generate audit log (timestamp, transitions)
      4. Compute statistics (% time per state, mean RMS, max RMS)
      5. Plot dashboard: signal + thresholds + state trajectory + alert events
    
    Parameters
    ----------
    rms_signal  : array-like, RMS velocity time series (mm/s)
    timestamps  : list of datetime, optional
    thresholds  : tuple (T_warn, T_alert, T_crit), default ISO 10816 Group 2
    hysteresis  : float, margin antara T_on dan T_off
    debounce_n  : int, jumlah consecutive samples untuk transition
    plot        : bool, generate dashboard plot
    
    Return: dict dengan state_history, audit_log, stats, fsm
    """
    rms = np.asarray(rms_signal)
    n = len(rms)
    if timestamps is None:
        ts_start = datetime(2025, 11, 15, 8, 0)
        timestamps = [ts_start + timedelta(seconds=30 * i) for i in range(n)]

    # Setup FSM dengan custom thresholds
    fsm = MonitoringFSM()
    fsm.THRESHOLDS = {'WARNING': thresholds[0], 'ALERT': thresholds[1], 'CRITICAL': thresholds[2]}
    fsm.HYSTERESIS = hysteresis
    fsm.DEBOUNCE_N = debounce_n

    # Run FSM
    state_history = []
    for r, ts in zip(rms, timestamps):
        state_history.append(fsm.update(r, ts))

    # Statistics
    from collections import Counter
    state_count = Counter(state_history)
    stats_dict = {
        'n_samples': n,
        'duration_hours': (timestamps[-1] - timestamps[0]).total_seconds() / 3600,
        'mean_rms': float(rms.mean()),
        'max_rms': float(rms.max()),
        'pct_per_state': {s: round(100 * state_count[s] / n, 2) for s in fsm.STATES},
        'n_transitions': len(fsm.audit_log),
    }

    # Reporting
    print("═══ MONITORING PIPELINE REPORT ═══")
    print(f"  Duration : {stats_dict['duration_hours']:.1f} jam ({stats_dict['n_samples']} samples)")
    print(f"  Mean RMS : {stats_dict['mean_rms']:.3f} mm/s")
    print(f"  Max RMS  : {stats_dict['max_rms']:.3f} mm/s")
    print(f"  Transitions: {stats_dict['n_transitions']}")
    print(f"\n  % time per state:")
    for s, pct in stats_dict['pct_per_state'].items():
        print(f"    {s:10s}: {pct:5.1f}%")

    # Audit log
    audit_df = pd.DataFrame(fsm.audit_log)

    # Dashboard plot
    if plot:
        fig, axes = plt.subplots(3, 1, figsize=(14, 9),
                                 gridspec_kw={'height_ratios': [3, 1, 2]})

        # Top: signal + thresholds + zone bands
        zone_colors = {'A': '#22c55e', 'B': '#06b6d4', 'C': '#f59e0b', 'D': '#ec4899'}
        axes[0].axhspan(0, thresholds[0], color='#22c55e', alpha=0.1)
        axes[0].axhspan(thresholds[0], thresholds[1], color='#06b6d4', alpha=0.1)
        axes[0].axhspan(thresholds[1], thresholds[2], color='#f59e0b', alpha=0.1)
        axes[0].axhspan(thresholds[2], rms.max()*1.1, color='#ec4899', alpha=0.1)
        axes[0].plot(range(n), rms, color='#1f2937', lw=0.8, alpha=0.8)
        for t, c, lbl in zip(thresholds, ['#06b6d4', '#f59e0b', '#ec4899'], ['WARN', 'ALERT', 'CRIT']):
            axes[0].axhline(t, color=c, linestyle='--', lw=1, label=f'{lbl}={t}')
        axes[0].set_ylabel('RMS (mm/s)'); axes[0].legend(loc='upper left')
        axes[0].set_title(f'Monitoring Dashboard | duration {stats_dict["duration_hours"]:.1f} jam | {stats_dict["n_transitions"]} transitions')
        axes[0].grid(alpha=0.25)

        # Mid: state trajectory
        state_idx = [fsm.STATES.index(s) if s in fsm.STATES else 0 for s in state_history]
        colors_state = ['#22c55e', '#06b6d4', '#f59e0b', '#ec4899', '#a855f7']
        axes[1].scatter(range(n), state_idx, c=[colors_state[i] for i in state_idx], s=3)
        axes[1].set_yticks(range(len(fsm.STATES))); axes[1].set_yticklabels(fsm.STATES)
        axes[1].set_ylabel('State'); axes[1].grid(alpha=0.25)

        # Bottom: % time per state (bar)
        states_present = [s for s in fsm.STATES if stats_dict['pct_per_state'][s] > 0]
        pcts = [stats_dict['pct_per_state'][s] for s in states_present]
        bar_colors = [colors_state[fsm.STATES.index(s)] for s in states_present]
        axes[2].barh(states_present, pcts, color=bar_colors)
        for i, p in enumerate(pcts):
            axes[2].text(p + 0.5, i, f'{p:.1f}%', va='center', fontweight='bold')
        axes[2].set_xlabel('% time'); axes[2].set_xlim(0, 105)
        axes[2].set_title('Time Distribution per State'); axes[2].grid(axis='x', alpha=0.25)

        plt.tight_layout(); plt.show()

    return {'state_history': state_history, 'audit_log': audit_df,
            'stats': stats_dict, 'fsm': fsm}

Sampling Rate & Nyquist: untuk mill stand 1500 RPM (25 Hz fundamental) dengan target capture BPFO sampai harmonic 5×, berapa sampling rate optimal?

›

Bearing SKF 23048 di mill stand: BPFO ≈ 4.2 × shaft frequency. Pada 1500 RPM = 25 Hz shaft → BPFO ≈ 105 Hz. Untuk diagnosa fault, kita butuh capture sampai 5th harmonic = 525 Hz. Per Nyquist criterion, f_sampling ≥ 2 × f_max = 1050 Hz minimum. Tetapi noise floor + anti-aliasing margin → industri pakai f_s = 5× margin. DAQ NI USB-6009 max 48 kHz total/4 channel = 12 kHz per channel. Dengan 24 channel pakai multiplexer 1 ms → max f_s ~10 kHz total (~400 Hz/channel kalau aksi serentak). Bagaimana pilih?

💬 Petunjuk: trade-off antara signal completeness (high f_s capture all harmonics) vs data volume (24 channel × 5 kHz × 24 jam = 100M sample/hari = 400 MB/hari = 150 GB/tahun) vs cost (NI cDAQ multi-channel simultaneous Rp 50 jt vs USB-6009 Rp 5 jt). Strategi industri: tiered sampling — high f_s (5 kHz) untuk on-demand burst (5 detik tiap 1 jam = 25k sample), low f_s (10 Hz) untuk continuous trending RMS. Plus anti-aliasing filter analog low-pass sebelum ADC wajib.
QUICK CHECK — Strategi sampling paling efektif untuk constraint multi-channel + budget?
Pakai f_s = 1 Hz untuk semua (ultra-cheap), abaikan high frequency analysis
Tiered sampling strategy: continuous low-rate (f_s = 10 Hz) untuk RMS trending + threshold check, plus burst high-rate (f_s = 5 kHz selama 5 detik tiap 1 jam) untuk FFT/envelope analysis on-demand. Pakai NI cDAQ-9171 + IEPE module 24-bit ADC simultaneous. Anti-aliasing filter analog 2 kHz low-pass wajib. Storage: ~150 GB/tahun, manageable di local NAS. Cost: Rp 50 jt one-time setup
Pakai f_s = 50 kHz untuk semua channel sepanjang waktu (overkill, miliaran sample/hari)
Pakai f_s = 100 Hz fixed (compromise, miss BPFO harmonic)
✅ Tepat! Tiered Sampling Strategy adalah industrial best practice. (1) Low-rate continuous (f_s = 10 Hz): hanya RMS calculation realtime, bandwidth tipis (10 Hz × 24 channel × 24 jam = 8.6M sample/hari = 35 MB/hari). Cukup untuk threshold check ISO 10816 + trend monitoring. (2) High-rate burst (f_s = 5 kHz, 5 detik tiap jam): capture full spectrum untuk FFT analysis. Bandwidth: 5000 × 24 × 5 sec × 24 hr = 14.4M sample/hari = ~60 MB/hari. (3) Trigger-based capture: kalau RMS continuous mode threshold breach → trigger burst capture untuk diagnostic detail. (4) Anti-aliasing filter: analog 2 kHz Butterworth low-pass sebelum ADC — wajib untuk hindari aliasing high-freq noise. (5) Hardware: NI cDAQ-9174 chassis + 9234 IEPE module (4-channel × 51.2 kS/s simultaneous) Rp 50 jt = workhorse industrial setup. (6) Storage: TimescaleDB (continuous) + Parquet file (burst snapshots) di NAS Synology Rp 15 jt. Pelajaran: jangan over-engineer dengan 50 kHz semua waktu — bandwidth/storage/cost tidak proportional dengan diagnostic value yang didapat.
❌ Periksa kembali. f_s = 1 Hz miss BPFO 105 Hz total. f_s = 50 kHz overkill (10 GB/hari, storage problem). f_s = 100 Hz miss harmonic. Tiered = optimal balance.

Hysteresis Margin Tuning: di environment harsh dengan noise std 0.15 mm/s, berapa hysteresis margin (T_on − T_off) yang optimal? Trade-off chattering vs sensitivity?

›

Lab measurement noise std 0.15 mm/s baseline (sensor IEPE + EMI dari motor 3 MW + vibrasi background dari mill stand sebelahnya). Untuk transition Zone B → C dengan T = 4.5 mm/s (ISO 10816 Group 3): margin terlalu kecil = chattering saat sinyal di sekitar 4.5 ± noise. Margin terlalu besar = miss subtle degradation, response lambat. Pertanyaan: bagaimana set hysteresis margin yang ideal?

Noise std: 0.15 mm/s Threshold: 4.5 mm/s (B→C) Schmitt trigger pattern Margin = 2–3× noise std
💬 Tidak ada jawaban tunggal — yang dinilai pemahaman trade-off. Pertimbangan: (1) Margin = 2–3× noise std aturan industrial standar. Untuk noise std 0.15 mm/s → margin 0.3–0.45 mm/s. Pakai T_on = 4.5, T_off = 4.5 − 0.4 = 4.1 mm/s. Chattering hilang karena noise (0.15 mm/s) < margin (0.4 mm/s). (2) Margin terlalu kecil (≤ noise): chattering — alarm ON/OFF/ON/OFF. Operator marah, ignore alarm penting. (3) Margin terlalu besar (≥ 1 mm/s): lambat respond ke recovery, jadi sinyal sehat lama dianggap masih alert. Plus miss subtle re-degradation. (4) Adaptive margin: rolling window noise estimation. Tiap 30 menit hitung std dari 30 menit data, set margin = 3× std current. Useful untuk environment yang noise level berubah (mis. cuaca, beban produksi). (5) Validation: monitor metric "transitions per hour" dan "time spent in transition" sebelum/sesudah set margin. Optimum saat transitions/hour ≤ 2 di steady-state. (6) Tier alarm + persistence timer di P12 melengkapi hysteresis untuk handle multi-tier escalation + auto re-evaluate.
