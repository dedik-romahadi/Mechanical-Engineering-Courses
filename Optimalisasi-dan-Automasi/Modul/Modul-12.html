Bagian 01

Mengapa Tier Alarm + FSM:
Production-Grade Alarm Management Architecture

Pertemuan 11 mengajarkan threshold tunggal + hysteresis. Cukup untuk lab demo, tetapi kurang untuk industrial production. Tiga problem yang muncul saat sistem real beroperasi 24/7: (1) Binary harshness — threshold tunggal hanya berikan OK / DANGER, tidak ada nuansa "hampir alarm" untuk preventive action. (2) No memory — sistem tidak ingat berapa lama mesin sudah di kondisi marginal. (3) Alarm flood — saat trip terjadi, puluhan alarm meledak bersama → operator overwhelmed → safety risk. Solusi production-grade: tier alarm escalation (Normal → Warning → Alert → Critical), finite state machine (FSM dengan memory + transition logic + audit log), dan compliance dengan ANSI/ISA 18.2 Alarm Management Standard. Pertemuan 12 fokus pada arsitektur ini — pondasi yang dipakai di setiap kilang minyak, pabrik kimia, pembangkit listrik di dunia. Setelah ini, P13–P14 lanjut ke ML predictive yang complement (bukan replace) rule-based foundation.

Lima Pilar Alarm Management Production-Grade

Gradual Escalation: 4-Tier vs Binary

Binary (OK / DANGER) = harsh + reactive. Tier 4-level: Normal → Warning → Alert → Critical = gradual + proactive. Tiap tier punya action berbeda: Warning = log + email; Alert = SMS supervisor; Critical = auto-shutdown + page operator. Operator bisa preventive action di Warning/Alert sebelum Critical. Memetakan ke ISO 10816 zona: A=Normal, B=Warning, C=Alert, D=Critical. Adopsi di Pertamina, Pupuk Kaltim, PLN sebagai SLA standard. Operator burnout turun, MTTR turun, false alarm efektif berkurang.

N → W → A → C (gradual)
🧠

FSM = Memory Sistem

Threshold sederhana stateless — tiap sample dievaluasi independen. FSM stateful — ingat: kapan masuk Warning, berapa lama persist, urutan transisi. Ini krusial untuk: (a) persistence-based escalation (Warning persist > 5 menit → escalate ke Alert), (b) auto-recovery dengan grace period (Alert → Normal hanya kalau persist normal > 1 menit), (c) audit trail (timestamp tiap transition untuk post-mortem analysis), (d) compliance reporting (durasi total mesin di tier C/D per bulan). Implementasi: 30 baris Python dengan statemachine library.

FSM: state + history + actions
⏳

Debouncing: Filter Noise Transient

Sensor noisy bisa cross threshold sesaat lalu kembali normal. Tanpa debouncing, satu spike → satu alarm → operator confused. Solusi: require N consecutive samples di kondisi alert sebelum trigger transition. N=3–5 standar untuk warning, N=1–2 untuk critical (low latency). Trade-off: N besar → robust noise tetapi delay deteksi naik. Pattern dari elektronics (button debouncing 50ms hardware), software equivalent untuk monitoring sensor. Implementasi: counter di-increment kalau kondisi true, reset kalau false. Ketika counter ≥ N, eksekusi transition.

if count ≥ N: transition

ANSI/ISA 18.2 — Alarm Management

Standar resmi ISA-18.2 (2009) — wajib di petrochemical, power, pulp&paper. Target: ≤ 1 alarm per 10 menit per operator di steady state. Lebih = alarm flood = operator overwhelmed = alarm fatigue → ignore alarm penting → safety risk (Texas City BP 2005, 15 fatalities partly due alarm flood). 7 lifecycle phases: identification, rationalization, documentation, implementation, operation, maintenance, monitoring & assessment. EEMUA 191 (UK) ekuivalen. Compliance audit cek alarm rate, response time, dan rationalization document.

≤ 1 alarm / 10 min

💡 Best Practice Alarm Management: (1) Mulai dengan rationalization — review semua alarm: apakah actionable? Apakah unique? Apakah priority benar? Hapus duplikat & non-actionable. (2) Tier dengan jelas — operator harus tahu di Warning vs Alert vs Critical, action berbeda. (3) Alarm rate target: ≤ 1/10 min steady, ≤ 10/10 min post-disturbance. Kalau lebih = ada masalah. (4) Time-based escalation — Warning persist > 5 menit → auto-escalate ke Alert. (5) Alarm shelving — operator boleh silence specific alarm untuk window terbatas (1–8 jam, mis. selama known maintenance). Otomatis re-enable. (6) Suppression rules — kalau parent alarm trigger (mis. "low oil pressure"), suppress child alarm yang correlated (mis. "high bearing temperature"). (7) Periodic alarm rate review — bulanan/quarterly. Alarm yang trigger > 100×/tahun tanpa actionable harus dihapus atau di-tune. (8) Operator training — alarm management hanya bekerja kalau operator paham filosofi tier escalation.
Bagian 02

Tier Alarm Architecture:
4-Level Escalation dari Normal hingga Critical

Tier alarm adalah arsitektur escalation gradual dari kondisi normal hingga critical, dengan action berbeda per tier. Bukan binary OK/FAIL — ada nuansa Warning + Alert sebelum Critical. Standar industri (ISA-18.2, EEMUA 191): 4 tier dengan severity meningkat dan response time menurun: (1) Normal — operasional sehat, no action. (2) Warning (Low Priority) — kondisi marginal, log + email engineer, monitor trend. (3) Alert (Medium Priority) — perlu intervensi, SMS supervisor + create work order. (4) Critical (High Priority) — bahaya immediate, auto-shutdown + page operator + escalate ke management. Setiap tier punya visual color coding (green/yellow/orange/red), response time SLA (none/4 hr/30 min/immediate), dan persistence rules (Warning > 5 min auto-escalate, Critical no recovery without manual ack).

Threshold Absolut: T_low & T_high

Bentuk paling sederhana: IF rms > T_high THEN ALERT. Untuk fitur dua-arah (mis. tekanan oli yang tidak boleh terlalu rendah dan terlalu tinggi): IF p < T_low OR p > T_high THEN ALERT. Threshold harus physical-meaningful: didasarkan pada batas mekanis (yield stress, bearing fatigue limit), standar industri, atau baseline operasi normal. Hindari threshold ditebak — selalu dokumentasikan sumbernya.

if x > T_high: alert
🏛️

ISO 10816-3 — Zone A/B/C/D

Standar internasional untuk vibrasi mesin rotasi. Pakai RMS velocity (mm/s) di band 10 Hz–1 kHz. Untuk Group 2 (medium 15–300 kW, rigid foundation): A (Good) ≤ 1.4, B (Acceptable) 1.4–2.8, C (Unsatisfactory) 2.8–4.5, D (Danger) > 4.5. Mesin di zona D harus shutdown atau diperbaiki segera. Klasifikasi otomatis = lookup table sederhana di Python: if rms ≤ 1.4: 'A' elif rms ≤ 2.8: 'B' elif rms ≤ 4.5: 'C' else: 'D'.

A | B | C | D zones
🔧

Class I–IV (Group Mesin)

Threshold ISO 10816-3 berbeda per kelas: Class I (small < 15 kW: motor pump kecil), Class II (medium 15–75 kW: blower, kompresor kecil), Class III (large machines on rigid foundations: turbin uap), Class IV (large machines on flexible foundations: paper machine, large compressor). Mesin yang sama bisa "OK" di Class IV (toleransi besar) tetapi "DANGER" di Class I (toleransi ketat). Selalu cek kelas mesin sebelum pilih threshold.

Class I (small) → IV (large flex)
🏗️

Rigid vs Flexible Foundation

Foundation = pondasi mesin. Rigid (concrete masif, baseplate steel tebal): vibrasi yang ditransmit kecil → threshold lebih ketat. Flexible (skid mounting, isolator karet, struktural baja): foundation menyerap vibrasi → threshold lebih longgar. ISO 10816 punya tabel terpisah untuk keduanya. Rule of thumb: kalau natural frequency foundation > 25% frequency mesin, considered rigid; kalau di bawah, flexible.

f_n_found vs f_machine

RMS Velocity 10 Hz – 1 kHz

ISO 10816 spesifik mengukur RMS velocity (bukan acceleration atau displacement) di band 10 Hz – 1 kHz. Pilihan ini berdasarkan: (1) velocity berkorelasi dengan energi kinetik vibrasi (≈ kerusakan), (2) band 10 Hz–1 kHz cover frekuensi mekanis dominan (running speed, gear meshing, bearing characteristic). Untuk mesin di luar range ini (mis. high-speed turbomachinery di 5–10 kHz), pakai standar lain (API 670: acceleration band > 5 kHz).

RMS_v = √(mean(v²)) di 10 Hz–1 kHz

Multi-Feature Threshold (RMS + Peak + Kurtosis)

Threshold tunggal (RMS) bisa miss anomali tertentu. Kombinasi multi-feature lebih sensitif: RMS deteksi unbalance/misalignment; peak amplitude deteksi impact (bearing pitting); kurtosis > 4 deteksi distribusi non-Gaussian (early bearing fault). Rule kombinasi: IF rms > T1 OR peak > T2 OR kurt > T3 THEN ALERT. Tools: P1–P7 (extract fitur) lalu P12 (apply rule). Kompleksitas grow → next step: ML (P13).

Bagian 03

Finite State Machine (FSM):
Memory + Transition Logic untuk Production Monitoring

Finite State Machine adalah model matematis formal yang sangat kuat untuk sistem stateful: didefinisikan oleh 5-tuple (S, s₀, Σ, δ, F) — set states S, initial state s₀, input alphabet Σ, transition function δ: S × Σ → S, dan final states F. Untuk monitoring tier alarm: S = {NORMAL, WARNING, ALERT, CRITICAL, FAULT}; s₀ = NORMAL; Σ = sensor reading + timer event; δ = transition rules dengan threshold + persistence + debounce. Dua varian: Moore machine (output bergantung HANYA pada state, lebih sederhana, predictable) vs Mealy machine (output bergantung state + input, lebih ekspresif tetapi kompleks). Untuk monitoring umum, Moore cukup. FSM memberikan: (1) memory — sistem ingat berapa lama di state, (2) transition logic — pindah antar tier dengan rules eksplisit, (3) audit trail — log setiap transition dengan timestamp untuk compliance + post-mortem. Implementasi: statemachine Python library (≤ 30 baris kode), atau pattern manual dengan dictionary state map.

Hysteresis: T_on > T_off

Definisi: dua threshold berbeda untuk transition ke "ON" vs ke "OFF". Saat status OFF, perlu sinyal > T_on untuk pindah ke ON. Saat status ON, sinyal harus turun di bawah T_off untuk kembali ke OFF. Tipikal: T_off = T_on − margin (mis. T_on=70, T_off=65). Margin 5–15% biasanya cukup. Tanpa hysteresis, sinyal di sekitar threshold ber-flip-flop sangat cepat (chattering).

T_on = 70, T_off = 65 (margin 5)
⚡

Schmitt Trigger Analogy

Konsep hysteresis dipakai di rangkaian elektronik Schmitt trigger (Otto Schmitt 1934) — comparator dengan dua threshold. Awalnya untuk filter noise di radar WWII, sekarang dipakai di setiap microcontroller (Arduino, STM32) untuk debouncing button input. Algoritma yang sama persis bisa diterapkan untuk threshold-based monitoring di software. Insight: setiap masalah noise-induced chattering punya solusi hysteresis yang elegan.

analog Schmitt trigger

Tier Alarm — Tiga Level Eskalasi

Bukan binary OK/FAIL, tetapi gradient 3+ level: WARNING (yellow, perhatikan) → ALERT (orange, action segera) → CRITICAL (red, shutdown). Tiap tier punya threshold sendiri: T_warn < T_alert < T_critical. Aksi berbeda: warning → log + notification email; alert → SMS ke supervisor; critical → auto-shutdown + page on-call engineer. Memetakan ke ISO 10816: A (no alarm), B (warning), C (alert), D (critical).

T_warn < T_alert < T_crit
⏱️

Escalation Timer (Persistence)

Beberapa anomali muncul sesaat (transient noise) dan menghilang sendiri — tidak perlu eskalasi. Tetapi kalau anomali persist > T detik, tier perlu naik. Contoh: warning persist 5 menit → eskalasi ke alert; alert persist 10 menit → eskalasi ke critical. Timer reset ketika sinyal kembali normal (atau hysteresis turun). Implementasi: state machine dengan timer per state.

if t_in_state > Δt: escalate

Alarm Flood Prevention (ANSI/ISA 18.2)

Bahaya nyata: alarm flood — saat 1 root cause memicu cascade puluhan alarm sekaligus, operator overwhelmed. ANSI/ISA 18.2 (Alarm Management) menetapkan limit: ≤ 1 alarm/10 menit per operator (steady state); ≤ 10 alarm dalam 10 menit pertama post-disturbance. Solusi: alarm grouping (suppress child alarm jika parent sudah on), shelving (operator bisa silence alarm selama window terbatas), alarm rationalization (review periodik untuk delete alarm yang tidak actionable).

≤ 1 alarm / 10 min

Konsistensi warna penting untuk operator yang shift-rotate antar mesin/pabrik. Gunakan ikon (✓ ⚠ ✗) di samping warna untuk accessibility (color-blind operators).

💡 Implementasi Hysteresis di Python (state machine sederhana):
class HysteresisAlarm:
    def __init__(self, t_on, t_off):
        self.t_on, self.t_off = t_on, t_off
        self.state = False
    def update(self, x):
        if not self.state and x > self.t_on: self.state = True
        elif self.state and x < self.t_off: self.state = False
        return self.state
Untuk tier alarm, extend ke 3 instance (warn/alert/crit) atau 1 class dengan list threshold. Frameworks production: OpenSCADA, Ignition, WinCC.
Bagian 04

Debouncing, Persistence & ANSI/ISA 18.2:
Filter Noise & Industrial Alarm Management Standard

Tier alarm + FSM butuh filtering tambahan supaya tidak chattering atau alarm flood. Dua mekanisme utama: debouncing (require N consecutive samples sebelum transition — filter noise transient yang bisa cross threshold sesaat lalu kembali normal) dan persistence timer (auto-escalate kalau warning persist > X menit, atau auto-recover kalau normal persist > Y menit). Plus, semua alarm management harus comply dengan ANSI/ISA 18.2 (2009) — standar industri global yang mengatur lifecycle alarm: identification → rationalization → documentation → implementation → operation → maintenance → monitoring & assessment. Target operational: ≤ 1 alarm per 10 menit per operator di steady state. Lebih = alarm flood = operator alarm fatigue = ignore alarm penting = safety hazard. Insiden Texas City Refinery (BP, 2005) — 15 fatalities, partly attributed to alarm flood. Kompliance ANSI/ISA 18.2 sekarang wajib di petrochemical, power, pharma di banyak negara.

States & Transitions

FSM didefinisikan oleh 5-tuple: (S, s₀, Σ, δ, F) — set states S, initial state s₀, input alphabet Σ, transition function δ: S × Σ → S, final states F. Untuk monitoring: S = {NORMAL, WARNING, ALERT, CRITICAL, FAULT}; s₀ = NORMAL; Σ = {sensor reading + timer event}; δ = rules (mis. NORMAL → WARNING jika rms > T_warn). Visualisasi via state diagram (lingkaran = state, panah = transition). Tools: graphviz, state_machine Python lib.

δ: S × Σ → S
🅼

Mealy vs Moore Machine

Dua varian FSM: Moore — output hanya bergantung pada state saat ini (mis. di state ALERT, selalu nyalakan lampu oranye, terlepas dari input). Mealy — output bergantung pada state + input (mis. transisi ALERT → CRITICAL kirim SMS; tetapi NORMAL → CRITICAL juga kirim SMS dengan pesan berbeda). Mealy lebih ekspresif tetapi rumit. Untuk monitoring sederhana, Moore cukup.

Moore: f(s); Mealy: f(s, σ)
🚪

Entry / Exit Actions

Tiap state punya 2 hook: on_entry (jalankan saat state baru di-enter) dan on_exit (jalankan saat state ditinggalkan). Contoh: on_entry(ALERT) = log timestamp + send SMS supervisor. on_exit(ALERT) = log durasi + cancel SMS escalation timer. Pola ini memisahkan logic transisi dari side effects (notifikasi, logging). Pattern populer: state pattern (Gang of Four), implementation di Python statemachine library.

on_entry / on_exit hooks
⏳

Debouncing (N Consecutive Samples)

Sinyal sensor noisy bisa cross threshold sesaat lalu kembali normal. Solusi: require N consecutive samples di kondisi alert sebelum pindah state. Contoh: untuk pindah dari NORMAL ke WARNING, butuh 5 sample berturut-turut dengan rms > T_warn. Implementasi: counter di-increment kalau kondisi true, reset ke 0 kalau false. Ketika counter >= N, eksekusi transisi. Trade-off: N besar → lebih robust noise, tetapi delay deteksi (latency) makin lama. Tipikal: N = 3–10 (≈ 100–500 ms pada 50 Hz sampling).

if count >= N: transition
💓

Heartbeat & Sensor Timeout

Sensor bisa mati / disconnect tanpa warning — output frozen di nilai terakhir. Untuk detect, FSM butuh heartbeat: harapkan sample baru tiap T detik. Kalau tidak ada sample selama 2·T (atau N·T), pindah ke state FAULT (sensor failure). Implementasi: timer yang reset tiap kali ada sample masuk; expired timer trigger transisi. Pattern: watchdog timer, dipakai di mikrokontroller (STM32 IWDG, AVR WDT) untuk detect software hang juga.

if Δt > timeout: FAULT
📜

Tiap transition state harus di-log dengan: timestamp, old_state, new_state, trigger value, sample data. Format JSON / CSV / database (SQLite, InfluxDB, TimescaleDB). Audit log = source of truth untuk: (a) post-mortem analysis (kenapa shutdown?), (b) compliance audit (ISO/API), (c) trend analysis (frequency of warnings naik = degradasi mesin), (d) ML training data (P13). Pattern: event sourcing — replay log untuk reconstruct state apapun di waktu lalu.

Operator inspect, ketemu coupling misalignment → fix → RMS turun 0.8 → 10 sample < T_off (1.2) selama 1 menit → ALERT → NORMAL (log recovery). Total event log = 4 entries — audit trail lengkap.
⚠️

Trap FSM & Debouncing yang Sering Diabaikan

(1) State explosion — kombinasi state dengan banyak feature (mis. RMS × peak × kurt × suhu) cepat tidak terkelola. Solusi: composite state atau hierarchical FSM (HFSM). (2) Race condition — sample baru tiba saat transition belum selesai. Pakai mutex / atomic update. (3) Stuck state — kalau exit condition tidak pernah tercapai (mis. T_off terlalu rendah), state stuck selamanya. Tambah escape route via timeout. (4) Debounce N terlalu besar — latency deteksi naik, alarm critical bisa terlambat. Trade-off: low N untuk critical (≤ 2), high N untuk warning (5–10). (5) Lupa heartbeat — sensor mati silent → output frozen → FSM tetap "NORMAL". Watchdog timer wajib. (6) Audit log overflow — high-frequency transitions bisa fill disk. Pakai log rotation, ringkasan (N events/min), atau database time-series (InfluxDB) yang punya retention policy. (7) State machine tanpa diagram — kode lama tanpa visualisasi sulit di-debug. Selalu maintain state diagram (graphviz) yang sinkron dengan code.

(2) Lupa hysteresis — hampir setiap newcomer engineer membangun rule tanpa hysteresis, lalu komplain "alert spam". Hysteresis = mandatory. (3) Single feature monitoring — RMS saja sering miss bearing fault (kurtosis lebih sensitif). Pakai multi-feature minimum 2 (RMS + peak atau RMS + kurtosis). (4) Tidak ada sensor health check — heartbeat & range validation (sensor reading di luar physically possible range = sensor failure, bukan reading valid). (6) Tidak ada alarm rationalization — review periodik (6–12 bulan): alarm yang tidak pernah actionable harus dihapus. (7) Threshold tidak adaptif drift — komponen mesin baru beda dari aged. Re-tune threshold setelah 6 bulan operational atau setelah major maintenance. Modern best practice: rule-based core + adaptive threshold via ML (Pertemuan 13).

Cell 2 — HysteresisAlarm Class: Hindari Chattering dengan Dual Threshold (T_on / T_off) Python

# ═══════════════════════════════════════════════════════════════════
# SCENARIO: Sinyal noisy di sekitar threshold (boundary case)
#   Tanpa hysteresis: chattering (alarm flicker)
#   Dengan hysteresis: stable transition
#   Demo: T_on = 2.8 (Zone B→C), T_off = 2.6 (margin 0.2)
# ═══════════════════════════════════════════════════════════════════

class HysteresisAlarm:
    """Threshold dengan hysteresis untuk hindari chattering."""
    def __init__(self, t_on, t_off, name='alarm'):
        assert t_on > t_off, "T_on harus > T_off untuk hysteresis"
        self.t_on = t_on
        self.t_off = t_off
        self.name = name
        self.state = False
        self.transitions = 0

    def update(self, x):
        if not self.state and x > self.t_on:
            self.state = True
            self.transitions += 1
        elif self.state and x < self.t_off:
            self.state = False
            self.transitions += 1
        return self.state

# Simple threshold (no hysteresis) untuk komparasi
class SimpleThreshold:
    def __init__(self, t):
        self.t = t
        self.state = False
        self.transitions = 0
    def update(self, x):
        new_state = x > self.t
        if new_state != self.state:
            self.transitions += 1
        self.state = new_state
        return self.state

# Generate sinyal di sekitar boundary (rms ≈ 2.7 ± 0.15 noise)
n = 500
t = np.arange(n)
rms = 2.7 + 0.15 * np.random.randn(n)        # Sinyal di sekitar threshold
rms[200:350] += 0.5                              # Genuine elevation di tengah

# Apply both alarms
hyst = HysteresisAlarm(t_on=2.8, t_off=2.6, name='WarningC')
simple = SimpleThreshold(t=2.8)

state_hyst = []
state_simple = []
for r in rms:
    state_hyst.append(hyst.update(r))
    state_simple.append(simple.update(r))

state_hyst = np.array(state_hyst, dtype=int)
state_simple = np.array(state_simple, dtype=int)

print("═══ COMPARISON: Simple Threshold vs Hysteresis ═══")
print(f"  Simple threshold transitions: {simple.transitions}  (chattering!)")
print(f"  Hysteresis transitions       : {hyst.transitions}  (stable)")
print(f"  Reduksi spam: {100*(1 - hyst.transitions/simple.transitions):.0f}%")

import pandas as pd
from datetime import datetime, timedelta

# ═══════════════════════════════════════════════════════════════════
# SCENARIO: Pump monitoring 12 jam, 1 sample / 30 detik = 1440 samples
#   FSM dengan 5 state: NORMAL → WARNING → ALERT → CRITICAL + FAULT
#   Debouncing: butuh N=3 consecutive samples untuk transition
#   Tier thresholds: T_warn=1.4, T_alert=2.8, T_crit=4.5 (ISO 10816)
#   Hysteresis: T_off = T_on - 0.2
#   Audit log: timestamp, old_state, new_state, trigger_value
# ═══════════════════════════════════════════════════════════════════

class MonitoringFSM:
    """Finite State Machine untuk monitoring tier alarm dengan debouncing."""

    STATES = ['NORMAL', 'WARNING', 'ALERT', 'CRITICAL', 'FAULT']
    THRESHOLDS = {'WARNING': 1.4, 'ALERT': 2.8, 'CRITICAL': 4.5}
    HYSTERESIS = 0.2
    DEBOUNCE_N = 3

    def __init__(self):
        self.state = 'NORMAL'
        self.up_counter = 0           # consecutive samples above threshold
        self.down_counter = 0         # consecutive samples below threshold
        self.audit_log = []
        self.last_sample_time = None

    def _target_state(self, x):
        """Tentukan state target berdasarkan threshold (tanpa hysteresis)."""
        if x >= self.THRESHOLDS['CRITICAL']:
            return 'CRITICAL'
        elif x >= self.THRESHOLDS['ALERT']:
            return 'ALERT'
        elif x >= self.THRESHOLDS['WARNING']:
            return 'WARNING'
        else:
            return 'NORMAL'

    def update(self, x, ts=None):
        """Process satu sample, return current state."""
        if ts is None:
            ts = datetime.now()
        self.last_sample_time = ts

        # Sensor health: kalau reading tidak masuk akal → FAULT
        if x < 0 or x > 50:
            self._transition('FAULT', x, ts, reason='Out-of-range sensor')
            return self.state

        target = self._target_state(x)
        target_idx = self.STATES.index(target)
        current_idx = self.STATES.index(self.state) if self.state != 'FAULT' else 0

        # Debouncing: increment counter only if direction consistent
        if target_idx > current_idx:
            self.up_counter += 1
            self.down_counter = 0
            if self.up_counter >= self.DEBOUNCE_N:
                self._transition(target, x, ts, reason='Escalation')
                self.up_counter = 0
        elif target_idx < current_idx:
            # Apply hysteresis: turun state hanya kalau x < T - hysteresis
            below_off = self._below_off_threshold(x, current_idx)
            if below_off:
                self.down_counter += 1
                self.up_counter = 0
                if self.down_counter >= self.DEBOUNCE_N:
                    self._transition(target, x, ts, reason='Recovery')
                    self.down_counter = 0
        else:
            self.up_counter = 0
            self.down_counter = 0

        return self.state

    def _below_off_threshold(self, x, current_idx):
        """Cek apakah x di bawah T_off (T - hysteresis) dari state saat ini."""
        if current_idx == 0:
            return False          # Already at NORMAL
        current_name = self.STATES[current_idx]
        t_off = self.THRESHOLDS[current_name] - self.HYSTERESIS
        return x < t_off

    def _transition(self, new_state, trigger, ts, reason=''):
        """Eksekusi state transition + log."""
        old_state = self.state
        self.state = new_state
        self.audit_log.append({
            'timestamp': ts,
            'old_state': old_state,
            'new_state': new_state,
            'trigger_value': round(trigger, 3),
            'reason': reason,
        })

def run_monitoring_pipeline(rms_signal, timestamps=None,
                            thresholds=(1.4, 2.8, 4.5),
                            hysteresis=0.2, debounce_n=3,
                            plot=True):
    """
    End-to-end rule-based monitoring pipeline.
    
    Steps:
      1. Apply ISO 10816 zone classification
      2. Apply hysteresis-aware FSM with debouncing
      3. Generate audit log (timestamp, transitions)
      4. Compute statistics (% time per state, mean RMS, max RMS)
      5. Plot dashboard: signal + thresholds + state trajectory + alert events
    
    Parameters
    ----------
    rms_signal  : array-like, RMS velocity time series (mm/s)
    timestamps  : list of datetime, optional
    thresholds  : tuple (T_warn, T_alert, T_crit), default ISO 10816 Group 2
    hysteresis  : float, margin antara T_on dan T_off
    debounce_n  : int, jumlah consecutive samples untuk transition
    plot        : bool, generate dashboard plot
    
    Return: dict dengan state_history, audit_log, stats, fsm
    """
    rms = np.asarray(rms_signal)
    n = len(rms)
    if timestamps is None:
        ts_start = datetime(2025, 11, 15, 8, 0)
        timestamps = [ts_start + timedelta(seconds=30 * i) for i in range(n)]

    # Setup FSM dengan custom thresholds
    fsm = MonitoringFSM()
    fsm.THRESHOLDS = {'WARNING': thresholds[0], 'ALERT': thresholds[1], 'CRITICAL': thresholds[2]}
    fsm.HYSTERESIS = hysteresis
    fsm.DEBOUNCE_N = debounce_n

    # Run FSM
    state_history = []
    for r, ts in zip(rms, timestamps):
        state_history.append(fsm.update(r, ts))

    # Statistics
    from collections import Counter
    state_count = Counter(state_history)
    stats_dict = {
        'n_samples': n,
        'duration_hours': (timestamps[-1] - timestamps[0]).total_seconds() / 3600,
        'mean_rms': float(rms.mean()),
        'max_rms': float(rms.max()),
        'pct_per_state': {s: round(100 * state_count[s] / n, 2) for s in fsm.STATES},
        'n_transitions': len(fsm.audit_log),
    }

    # Reporting
    print("═══ MONITORING PIPELINE REPORT ═══")
    print(f"  Duration : {stats_dict['duration_hours']:.1f} jam ({stats_dict['n_samples']} samples)")
    print(f"  Mean RMS : {stats_dict['mean_rms']:.3f} mm/s")
    print(f"  Max RMS  : {stats_dict['max_rms']:.3f} mm/s")
    print(f"  Transitions: {stats_dict['n_transitions']}")
    print(f"\n  % time per state:")
    for s, pct in stats_dict['pct_per_state'].items():
        print(f"    {s:10s}: {pct:5.1f}%")

    # Audit log
    audit_df = pd.DataFrame(fsm.audit_log)

    # Dashboard plot
    if plot:
        fig, axes = plt.subplots(3, 1, figsize=(14, 9),
                                 gridspec_kw={'height_ratios': [3, 1, 2]})

        # Top: signal + thresholds + zone bands
        zone_colors = {'A': '#22c55e', 'B': '#06b6d4', 'C': '#f59e0b', 'D': '#ec4899'}
        axes[0].axhspan(0, thresholds[0], color='#22c55e', alpha=0.1)
        axes[0].axhspan(thresholds[0], thresholds[1], color='#06b6d4', alpha=0.1)
        axes[0].axhspan(thresholds[1], thresholds[2], color='#f59e0b', alpha=0.1)
        axes[0].axhspan(thresholds[2], rms.max()*1.1, color='#ec4899', alpha=0.1)
        axes[0].plot(range(n), rms, color='#1f2937', lw=0.8, alpha=0.8)
        for t, c, lbl in zip(thresholds, ['#06b6d4', '#f59e0b', '#ec4899'], ['WARN', 'ALERT', 'CRIT']):
            axes[0].axhline(t, color=c, linestyle='--', lw=1, label=f'{lbl}={t}')
        axes[0].set_ylabel('RMS (mm/s)'); axes[0].legend(loc='upper left')
        axes[0].set_title(f'Monitoring Dashboard | duration {stats_dict["duration_hours"]:.1f} jam | {stats_dict["n_transitions"]} transitions')
        axes[0].grid(alpha=0.25)

        # Mid: state trajectory
        state_idx = [fsm.STATES.index(s) if s in fsm.STATES else 0 for s in state_history]
        colors_state = ['#22c55e', '#06b6d4', '#f59e0b', '#ec4899', '#a855f7']
        axes[1].scatter(range(n), state_idx, c=[colors_state[i] for i in state_idx], s=3)
        axes[1].set_yticks(range(len(fsm.STATES))); axes[1].set_yticklabels(fsm.STATES)
        axes[1].set_ylabel('State'); axes[1].grid(alpha=0.25)

        # Bottom: % time per state (bar)
        states_present = [s for s in fsm.STATES if stats_dict['pct_per_state'][s] > 0]
        pcts = [stats_dict['pct_per_state'][s] for s in states_present]
        bar_colors = [colors_state[fsm.STATES.index(s)] for s in states_present]
        axes[2].barh(states_present, pcts, color=bar_colors)
        for i, p in enumerate(pcts):
            axes[2].text(p + 0.5, i, f'{p:.1f}%', va='center', fontweight='bold')
        axes[2].set_xlabel('% time'); axes[2].set_xlim(0, 105)
        axes[2].set_title('Time Distribution per State'); axes[2].grid(axis='x', alpha=0.25)

        plt.tight_layout(); plt.show()

    return {'state_history': state_history, 'audit_log': audit_df,
            'stats': stats_dict, 'fsm': fsm}

✅ Tepat! Hybrid Fixed + Adaptive Threshold adalah pola production-grade. (1) Fixed ISO threshold: untuk compliance (insurance, PSM regulation OSHA, audit ISO 55000), legal protection — jangan diubah.
