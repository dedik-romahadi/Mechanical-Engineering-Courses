Bagian 01

Mengapa Linear Programming:
Keputusan Optimal di Tengah Sumber Daya Terbatas

Setiap insinyur mesin pasti menghadapi pertanyaan klasik: berapa banyak unit A vs unit B yang harus diproduksi agar profit maksimum? atau komposisi material mana yang memberi kekuatan tertinggi dengan biaya terendah? Tanpa metode formal, jawabannya hanya tebakan. Linear Programming (LP) memberi kerangka matematis: rumuskan tujuan sebagai fungsi linier Z = cᵀx, batasi solusi dengan kendala linier Ax ≤ b dan x ≥ 0, lalu cari titik di daerah feasibel yang memaksimalkan/meminimalkan Z. Dalle (1947), George Dantzig membuktikan: solusi optimum LP selalu berada di vertex (titik sudut) daerah feasibel — sehingga pencarian brute force O(2ⁿ) bisa diturunkan jadi algoritma Simpleks yang efisien.

Lima Konsep Inti Linear Programming
Variabel keputusan x  ·  Fungsi objektif cᵀx  ·  Kendala linier Ax ≤ b  ·  Daerah feasibel (poligon konveks)  ·  Titik optimal di vertex
📐

Linearitas — Asumsi Fundamental

LP mengasumsikan linearitas di tiga level: (1) fungsi objektif linier — profit per unit konstan; (2) kendala linier — pemakaian sumber daya proporsional; (3) aditivitas — total = jumlah individu, tanpa interaksi sinergis. Asumsi ini valid untuk banyak masalah skala produksi (short-term, sumber daya tetap). Untuk fungsi non-linier (skala ekonomi, biaya kuadratik), gunakan NLP di Pertemuan 10.

Z = c₁x₁ + c₂x₂ + ... + cₙxₙ
🔺

Daerah Feasibel = Poligon Konveks

Setiap kendala aᵢᵀx ≤ bᵢ adalah setengah-bidang. Irisan semua kendala membentuk polihedron konveks. Sifat konveks ini krusial: setiap maksimum/minimum lokal sekaligus global — tidak ada "perangkap" lokal seperti di NLP. Inilah yang membuat LP tractable bahkan untuk ratusan ribu variabel.

P = {x | Ax ≤ b, x ≥ 0}
🎯

Teorema Dasar LP (Dantzig 1947)

Jika LP punya solusi optimal terbatas, maka setidaknya satu titik vertex daerah feasibel adalah solusi optimal. Konsekuensi: alih-alih cek semua titik di poligon (jumlahnya tak hingga), cukup periksa vertex saja — jumlahnya finite (paling banyak C(m+n, n)). Algoritma Simpleks melompat dari vertex ke vertex, selalu memperbaiki Z, hingga tidak ada peningkatan lagi.

x* ∈ vertices(P)
🐍

Implementasi: scipy.optimize.linprog

Python menyediakan solver LP gratis lewat scipy.optimize.linprog. Cukup definisikan: c (koefisien objektif), A_ub/b_ub (kendala ≤), A_eq/b_eq (kendala =), bounds (batas variabel). Method default 'highs' (HiGHS solver, 2020) menyelesaikan masalah hingga jutaan variabel dalam detik. Untuk LP industrial: pakai PuLP/Pyomo + Gurobi/CPLEX.

linprog(c, A_ub, b_ub, ...)

Bagian 02

Formulasi Standar:
Bentuk Matematis Linear Programming

Setiap LP dapat ditulis dalam bentuk standar agar dapat diselesaikan oleh solver. Ada dua konvensi populer: (1) bentuk standar maksimisasi dengan kendala ≤ dan variabel non-negatif; (2) bentuk standar minimisasi dengan kendala = dan slack/surplus variable. Memahami konversi antar bentuk adalah kunci agar masalah riil bisa di-feed ke solver. Setiap masalah LP dapat dibawa ke salah satu bentuk standar lewat tiga manuver sederhana: substitusi variabel bebas, negasi kendala ≥, dan introduksi slack/surplus.

Empat Komponen Wajib Formulasi LP
Variabel x ∈ ℝⁿ  ·  Objektif min/max cᵀx  ·  Kendala Ax ≤ b (atau =)  ·  Bounds x ≥ 0 (atau lb ≤ x ≤ ub)
📝

Bentuk Kanonik (Maksimisasi)

Bentuk paling intuitif untuk masalah produksi: max Z = cᵀx dengan kendala Ax ≤ b dan x ≥ 0. Setiap baris aᵢᵀx ≤ bᵢ merepresentasikan batas sumber daya (kapasitas mesin, bahan baku, jam kerja). Inequality ≤ adalah default karena menggambarkan "tidak boleh melebihi". Untuk minimisasi, kalikan c dengan −1.

max cᵀx s.t. Ax ≤ b, x ≥ 0
🔄

Bentuk Standar (Slack Variables)

Konversi aᵢᵀx ≤ bᵢ menjadi persamaan dengan menambah slack variable sᵢ ≥ 0: aᵢᵀx + sᵢ = bᵢ. Slack mewakili sumber daya yang tidak terpakai. Untuk kendala ≥, kurangi surplus variable. Hasil: sistem persamaan linier [A | I]·[x; s] = b dengan semua variabel non-negatif — bentuk yang dipakai algoritma Simpleks.

Ax + Is = b, x,s ≥ 0
🎚️

Bounds — Variabel Berbatas

Variabel kadang punya batas atas (kapasitas maksimum produksi) dan batas bawah (kontrak minimum). Daripada menulis ulang sebagai kendala umum, solver modern menerima langsung: lb ≤ x ≤ ub. Di SciPy: bounds=[(0, 100), (0, None), ...]. Untuk variabel bebas (tanpa tanda), pakai (None, None) — internal akan dipecah menjadi x = x⁺ − x⁻.

lb ≤ x ≤ ub
⚠️

Tiga Status Solusi LP

Saat solver berjalan, hasilnya salah satu dari: (1) Optimal — solusi unik atau multiple optima ditemukan; (2) Infeasible — kendala kontradiktif, daerah feasibel kosong; (3) Unbounded — Z bisa naik tanpa batas (formulasi kurang kendala). Selalu cek result.status sebelum percaya result.x — jangan asumsi optimal.

status ∈ {0, 2, 3}
Contoh Konkret — Pabrik Komponen Mesin:
Pabrik menghasilkan dua jenis bracket: A (profit Rp 40k/unit) dan B (profit Rp 30k/unit). Kendala: mesin CNC (60 jam tersedia, A pakai 2 jam, B pakai 1 jam) dan welding (80 jam, A pakai 1 jam, B pakai 3 jam). Formulasi LP: max Z = 40x_A + 30x_B s.t. 2x_A + x_B ≤ 60, x_A + 3x_B ≤ 80, x_A, x_B ≥ 0. Solusi optimal di vertex (20, 20) dengan Z = Rp 1.400k.
💡 Aturan Konversi (Cheat Sheet): (1) Min ↔ Max: min cᵀx ≡ max −cᵀx (negasi nilai akhir, tidak mengubah x). (2) ≥ jadi ≤: kalikan kedua sisi dengan −1. (3) = jadi dua ≤: tulis sebagai aᵀx ≤ b dan −aᵀx ≤ −b. (4) x bebas: ganti dengan x = x⁺ − x⁻, dengan x⁺, x⁻ ≥ 0. (5) Selalu cek dimensi A, b, c sebelum solve — banyak bug LP berasal dari salah orientasi matrix.
Bagian 03

Daerah Feasibel & Metode Grafis:
Geometri di Balik Linear Programming

Untuk LP 2 variabel (x₁, x₂), kita bisa menyelesaikan masalah dengan plot di kertas grafik. Setiap kendala adalah garis lurus; sisi yang memenuhi kendala adalah setengah-bidang. Irisan semua setengah-bidang membentuk daerah feasibel — biasanya berupa poligon konveks. Lalu plot garis kontur fungsi objektif cᵀx = const dan geser ke arah gradien c hingga menyentuh vertex terjauh. Vertex itulah solusi optimal. Metode grafis ini tidak praktis untuk n ≥ 3, tapi sangat penting untuk membangun intuisi geometris sebelum masuk ke Simpleks.

Empat Komponen Geometris LP
Constraint line: aᵢᵀx = bᵢ  ·  Half-plane: aᵢᵀx ≤ bᵢ  ·  Vertex (extreme point)  ·  Iso-profit line: cᵀx = c₀
📈

Plot Constraint Lines

Untuk setiap kendala a₁x₁ + a₂x₂ ≤ b: gambar garis batas a₁x₁ + a₂x₂ = b dengan dua titik (potong sumbu x₁ saat x₂=0, dan sumbu x₂ saat x₁=0). Lalu uji satu titik (biasanya origin) — jika memenuhi kendala, sisi origin adalah feasible; jika tidak, sisi sebaliknya. Ulangi untuk semua kendala. Kendala x ≥ 0 membatasi ke kuadran pertama.

x₁/b₁ + x₂/b₂ = 1
🔻

Identifikasi Daerah Feasibel

Daerah feasibel = irisan semua setengah-bidang. Berbentuk poligon konveks (mungkin bounded atau unbounded). Vertex daerah feasibel = perpotongan dua atau lebih garis constraint yang aktif (binding). Untuk LP 2D dengan m kendala + 2 non-negativitas, jumlah vertex ≤ C(m+2, 2). Salah satu vertex inilah solusi optimal (Teorema Dasar LP).

P = ⋂ {x | aᵢᵀx ≤ bᵢ}
🎯

Iso-Profit Line & Optimasi

Plot garis c₁x₁ + c₂x₂ = Z₀ untuk Z₀ kecil. Garis ini punya kemiringan tetap −c₁/c₂; nilai Z naik saat garis bergeser ke arah vektor gradien c. Geser garis menjauhi origin (untuk maksimisasi) hingga menyentuh vertex terakhir daerah feasibel. Vertex itu = solusi optimal x*. Jika garis sejajar dengan satu sisi → multiple optima.

∇Z = c → arah peningkatan
🔍

Enumerasi Vertex (Brute Force)

Alternatif metode grafis: cari semua vertex, evaluasi Z di tiap vertex, pilih yang terbesar. Vertex = solusi sistem persamaan dari 2 kendala aktif. Untuk n=2 dengan m kendala: cek semua C(m,2) kombinasi pasangan, validasi feasibility, lalu evaluasi Z. Cocok untuk verifikasi manual hasil solver di kasus kecil.

x* = argmax{Z(v) : v ∈ vertices}

Bagian 04

Metode Simpleks:
Algoritma Klasik Penyelesai LP

Diciptakan oleh George Dantzig (1947) selama Perang Dunia II untuk USAF, algoritma Simpleks adalah metode pertama yang membuat LP tractable. Idenya elegan: mulai dari salah satu vertex daerah feasibel, periksa apakah ada vertex tetangga yang memberi nilai objektif lebih baik, dan jika ya — pivot ke vertex itu. Ulangi sampai tidak ada perbaikan. Karena daerah feasibel konveks, maksimum lokal = maksimum global. Meski worst-case eksponensial (Klee–Minty 1972), dalam praktik Simpleks selesai dalam ≈ 2m–3m iterasi untuk masalah dengan m kendala — sangat efisien.

Empat Tahap Algoritma Simpleks
Initial BFS  ·  Pricing (reduced cost zⱼ − cⱼ)  ·  Ratio test (min bᵢ/aᵢⱼ)  ·  Pivot (Gauss-Jordan)
🔢

Tableau Awal & Basic Feasible Solution

Konversi LP ke bentuk standar dengan slack, lalu susun tableau: baris kendala + baris objektif. Basic variables (BV) = kolom yang membentuk identity (awalnya semua slack); non-basic (NBV) = sisanya, di-set ke 0. Solusi awal: x = 0, slack = b → titik origin di daerah feasibel (asumsi b ≥ 0).

x_BV = B⁻¹b, x_NBV = 0
💰

Pricing — Pilih Variabel Masuk

Hitung reduced cost tiap NBV: z̄ⱼ = cⱼ − c_BV·B⁻¹·aⱼ. Untuk maksimisasi: pilih NBV dengan z̄ⱼ paling positif (Dantzig's rule) — variabel ini akan masuk basis dan menaikkan Z. Jika semua z̄ⱼ ≤ 0 → solusi optimal, stop. Alternatif: steepest edge (lebih cepat tapi lebih mahal per iterasi), Bland's rule (anti-cycling).

j* = argmax{z̄ⱼ : NBV}
📏

Ratio Test — Pilih Variabel Keluar

Setelah variabel x_{j*} dipilih masuk, harus ada satu BV yang keluar (agar tetap n basic variables). Minimum ratio test: untuk semua baris i dengan aᵢⱼ* > 0, hitung θᵢ = bᵢ / aᵢⱼ*. Pilih i* dengan θ minimum — itulah BV yang keluar. Jika semua aᵢⱼ* ≤ 0 → unbounded (Z bisa naik ∞).

θ* = min{bᵢ/aᵢⱼ* : aᵢⱼ* > 0}
🔄

Pivot — Update Tableau

Pivot di posisi (i*, j*): bagi baris i* dengan aᵢ*ⱼ* agar pivot = 1, lalu eliminasi kolom j* di semua baris lain dengan row operations Gauss-Jordan. Ulangi pricing → ratio test → pivot. Tiap iterasi: Z monoton naik (untuk maks). Konvergen dalam jumlah finite langkah karena vertex jumlahnya finite, kecuali ada degenerasi (cycling).

Z_(k+1) ≥ Z_k
⚡

Big-M & Two-Phase (Initial BFS)

Saat tidak ada slack/identitas siap (mis. ada kendala = atau ≥), pakai artificial variable. Dua pendekatan: (1) Big-M method: tambah artificial dengan koefisien penalti M besar di objektif → solver akan menekan ke 0; (2) Two-Phase method: minimize sum of artificial dulu (Phase 1), lalu solve LP asli (Phase 2). Two-Phase lebih numerik stabil di praktik.

min M·Σaⱼ + cᵀx
🚀

Revised Simplex & Modern Solver

Tableau klasik boros memori (matrix m×(n+m)). Revised Simplex hanya menyimpan basis B dan inverse-nya — hemat memori, lebih cepat untuk LP besar. Modern solver: HiGHS (default scipy 1.9+), Gurobi, CPLEX kombinasi Simpleks + Interior Point Method (Karmarkar 1984, polynomial worst-case). Untuk LP n > 10⁶: pakai HiGHS dual simplex.

Dual Simplex & Sensitivity Analysis:
Setiap LP primal punya dual yang setara. Dual variable y menunjukkan shadow price tiap kendala — berapa peningkatan Z kalau bᵢ ditambah 1 unit. Misal pabrik dengan kendala 60 jam CNC dan 80 jam welding (lihat C2), solve sistem dual 2y₁+y₂=40, y₁+3y₂=30 → dual y_CNC = 18, y_welding = 4. Tambahan 1 jam CNC akan menaikkan profit Rp 18k (lebih besar dari welding). Insight bisnis: y memberi prioritas investasi — investasi di CNC memberi ROI lebih besar. Hasil scipy.optimize.linprog mengembalikan -result.ineqlin.marginals = nilai dual.
⚠️

Degenerasi & Cycling

Saat vertex degenerate (3+ kendala bertemu di 1 titik di 2D, atau k+1 kendala di kD), beberapa BV bernilai 0 → ratio test bisa tied → algoritma cycling (kembali ke vertex yang sama). Solusinya: Bland's rule (pilih j dengan index terkecil), lexicographic ordering, atau perturbation kecil di b. Solver modern (HiGHS) sudah menangani ini secara internal — cukup laporkan jika Anda lihat status=4 (numerical issue).

Analogi
Optimasi Linear di Kehidupan Sehari-hari

Linear Programming bukan konsep abstrak yang hanya hidup di buku teks — Anda sebenarnya melakukannya setiap hari, hanya tanpa menulis matrix-nya. Enam analogi di bawah ini menunjukkan bagaimana memilih kombinasi terbaik di tengah keterbatasan adalah keputusan LP yang muncul di mana-mana, dari diet harian hingga distribusi paket logistik global.

🍱 Analogi 1 — Diet Plan & Stigler's Diet Problem Pada 1939, ekonom George Stigler menanyakan: berapa biaya minimum makanan agar memenuhi semua kebutuhan gizi (kalori, protein, vitamin, mineral)? Ini adalah masalah LP klasik: variabel = jumlah tiap jenis makanan, objektif = minimize total biaya, kendala = setiap nutrisi harus ≥ recommended daily intake. Solusi Stigler manual (1939) memberi $39.93/tahun. Setelah Simpleks ditemukan (1947), Dantzig menyelesaikan ulang dalam menit dan dapat $39.69 — Stigler salah $0.24! Inilah motivasi awal LP: jawaban yang terbukti optimal, bukan tebakan terbaik.
🚚 Analogi 2 — Logistik & Transportation Problem Bagaimana Amazon menentukan dari gudang mana paket Anda dikirim? Ini transportation LP: variabel xᵢⱼ = jumlah unit dari gudang i ke kota j, objektif minimize total ongkos kirim, kendala = kapasitas gudang dan demand kota. Network simplex (variant LP) menyelesaikan masalah ini di skala miliaran paket per tahun. Sama persis dengan distribusi spare-part mesin: tentukan dari warehouse mana suku cadang dikirim ke pabrik agar total cost (transport + holding) minimum.
⛽ Analogi 3 — Refinery Blending & Petroleum LP Setiap kilang minyak menjalankan LP setiap jam. Crude oil punya beragam fraksi (bensin, solar, avtur, aspal). Berapa banyak tiap fraksi diproduksi agar profit maksimum, dengan kendala kualitas (oktan, sulfur, viskositas) dan demand pasar? Royal Dutch Shell mempelopori LP industri pada 1950-an — diperkirakan menghemat miliaran dolar. Variabel: laju aliran tiap unit; objektif: maximize Σ profit·flow; kendala: spesifikasi produk linier (mostly), kapasitas reaktor, balance massa. Tanpa LP, kilang hanya menebak.
📅 Analogi 4 — Penjadwalan Mesin di Pabrik Pabrik dengan 10 mesin CNC dan 50 produk. Mesin mana mengerjakan produk apa, kapan, agar throughput maksimal? Variabel: xᵢⱼₜ = produk i di mesin j di slot waktu t; objektif: max output (atau min idle time); kendala: kapasitas mesin tiap slot, urutan operasi (precedence), maintenance windows. Inilah inti Production Planning & Scheduling di MES. Kombinasi LP + integer programming (untuk variabel binary "mesin running/idle") menjadi MILP — bagian penting Industry 4.0.
💰 Analogi 5 — Portfolio & Markowitz's Model Investor mengalokasikan dana ke saham, obligasi, emas, kripto — agar return maksimum dengan risk tertentu. Markowitz (1952) rumuskan ini sebagai LP/QP: variabel wᵢ = bobot tiap aset, objektif = maximize Σwᵢ·rᵢ, kendala Σwᵢ = 1 (full investment), w ≥ 0 (no short), wᵀΣw ≤ σ²_target (risk budget). Setiap robo-advisor (Wealthfront, Bibit) menjalankan variant LP/QP ini saat Anda klik "rebalance". Konsep yang sama dipakai di pabrik: alokasi budget maintenance ke berbagai mesin agar reliability portfolio maksimum.
⚡ Analogi 6 — Power Grid & Economic Dispatch Saat 100+ pembangkit listrik perlu memenuhi demand grid 24/7, operator menjalankan LP setiap 5 menit. ⚠️

Jebakan Praktis Formulasi LP

(1) Lupa kendala non-negativitas x ≥ 0 → solver bisa pilih nilai negatif yang nonsens fisik (mis. produksi negatif). (2) Asumsi linier yang salah — biaya overtime ≠ linier, skala ekonomi ≠ linier; LP akan memberi solusi optimistik. (3) Unbounded karena kurang kendala — kalau LP unbounded, biasanya ada kendala fisik yang lupa ditulis (mis. kapasitas mesin tak terbatas). (4) Salah orientasi matrix A — pastikan A.shape = (m, n) dengan m = jumlah kendala, n = jumlah variabel. (5) Numerical scaling — koefisien 10⁻³ dan 10⁶ di matrix yang sama → kondisional buruk; normalisasi dulu. (6) Integer solutions diperlukan tapi pakai LP saja → solusi pecahan tidak realistis (mis. 3.7 truk); pakai MILP dengan variabel integer.

# ═══════════════════════════════════════════════════════════════════
# PROBLEM: Pabrik bracket logam memproduksi 2 model (A & B)
#   - Bracket A: profit Rp 40k/unit, butuh 2 jam CNC + 1 jam welding
#   - Bracket B: profit Rp 30k/unit, butuh 1 jam CNC + 3 jam welding
#   - Kapasitas: 60 jam CNC/minggu, 80 jam welding/minggu
#   - Tujuan: tentukan x_A & x_B agar profit maksimum
# ═══════════════════════════════════════════════════════════════════

# Variabel keputusan: x = [x_A, x_B]
# Fungsi objektif: max Z = 40·x_A + 30·x_B
# scipy.linprog MEMINIMUMKAN, jadi negasikan koefisien profit
c = [-40, -30]                              # minimize -Z = max Z

# Kendala bentuk Ax ≤ b
#   2·x_A + 1·x_B ≤ 60   (CNC)
#   1·x_A + 3·x_B ≤ 80   (welding)
A_ub = [[2, 1],
        [1, 3]]
b_ub = [60, 80]

# Bounds: x_A, x_B ≥ 0 (non-negativitas)
bounds = [(0, None), (0, None)]

# Solve dengan HiGHS (default modern, sangat cepat)
result = linprog(c, A_ub=A_ub, b_ub=b_ub, bounds=bounds, method='highs')

# Cek status & print solusi
print(f"Status: {result.message}")
print(f"Konvergen optimal: {result.success}")

if result.success:
    x_A, x_B = result.x
    Z_max    = -result.fun                       # negasikan kembali
    print(f"\n═══ SOLUSI OPTIMAL ═══")
    print(f"  x_A (bracket A)  = {x_A:8.2f} unit/minggu")
    print(f"  x_B (bracket B)  = {x_B:8.2f} unit/minggu")
    print(f"  Profit maks. Z   = Rp {Z_max*1000:>12,.0f}/minggu")

import pandas as pd

def solve_production_lp(profit, A_ub, b_ub, names=None, constraints=None):
    """Solver LP produksi reusable.
    profit       : list[float] — profit per unit tiap produk (max obj)
    A_ub, b_ub   : kendala linier ≤
    names        : label produk; constraints: label kendala
    Return dict { 'status', 'Z', 'production', 'shadow_prices', 'utilization' }
    """
    n = len(profit); m = len(b_ub)
    names       = names or [f'P{i+1}' for i in range(n)]
    constraints = constraints or [f'C{i+1}' for i in range(m)]

    c = [-p for p in profit]
    res = linprog(c, A_ub=A_ub, b_ub=b_ub,
                  bounds=[(0, None)]*n, method='highs')

    if not res.success:
        return {'status': res.message, 'Z': None}

    # Hitung pemakaian sumber daya tiap kendala
    A_arr = np.array(A_ub, dtype=float)
    used  = A_arr @ res.x
    util  = used / np.array(b_ub, dtype=float)

    return {
        'status'      : res.message,
        'Z'           : -res.fun,
        'production'  : pd.Series(res.x, index=names, name='unit'),
        'shadow_prices': pd.Series(-res.ineqlin.marginals, index=constraints, name='shadow'),
        'utilization' : pd.Series(util*100, index=constraints, name='used_%'),
        'slack'       : pd.Series(res.ineqlin.residual, index=constraints, name='slack'),
    }

# Pakai untuk masalah Cell 3 — fungsi reusable
out = solve_production_lp(
    profit=[30, 50, 40, 60],
    A_ub=[[2,3,2.5,4], [1,2,1.5,3], [0.5,1,1,2],
          [5,8,6,10], [0,-1,0,0], [0,0,0,1]],
    b_ub=[200, 150, 120, 800, -5, 30],
    names      =['P1', 'P2', 'P3', 'P4'],
    constraints=['CNC', 'Welding', 'Painting', 'Material', 'P2_min', 'P4_max'],
)

print(f"Status     : {out['status']}")
print(f"Z optimal  : Rp {out['Z']*1000:>12,.0f}/minggu\n")
print("═══ PRODUKSI OPTIMAL ═══")
print(out['production'].round(2).to_string())
print("\n═══ SHADOW PRICES, UTILISASI & SLACK ═══")

📋 KASUS INDUSTRI

Sebuah pabrik komponen otomotif Tier-1 di Cikarang memproduksi 4 jenis bracket logam (B1, B2, B3, B4) untuk supply ke OEM (Toyota, Honda, Daihatsu). Operations manager perlu menentukan mix produksi mingguan yang memberi profit maksimum. Data dari ERP system & cost accounting:

Welding: 150 jam/minggu (BOTTLENECK!)
Painting: 120 jam/minggu, Material: 800 kg/minggu
Demand kontrak: B2 ≥ 5 unit, B4 ≤ 30 unit/minggu
Solver LP: Z* ≈ Rp 2,737k/minggu

Pertanyaan kritis: hasil LP menunjukkan x = (0, 18.75, 0, 30) dengan B1 dan B3 = 0 unit (tidak diproduksi sama sekali). Pilihan investasi: (A) overtime welding 20 jam/minggu @ Rp 8k/jam tambahan biaya, (B) beli mesin welding kedua (Rp 800 juta, +60 jam/minggu kapasitas), (C) outsource sebagian welding ke vendor @ Rp 12k/jam. Interpretasi: Mengapa B1 dan B3 sama sekali tidak diproduksi (x₁ = x₃ = 0) padahal profitnya positif?

›

Solver LP memberi solusi optimal x = (0, 18.75, 0, 30) — produk B1 dan B3 tidak diproduksi sama sekali. Apakah ini kesalahan formulasi? Atau memang B2 dan B4 dominan secara opportunity cost di tiap kendala? Analisis dari perspektif reduced cost atau "rasio profit per pemakaian sumber daya bottleneck (welding)".

profit B1=30, welding B1=1 → rasio 30 profit B2=50, welding B2=2 → rasio 25 profit B4=60, welding B4=3 → rasio 20

✅ Tepat! Karakteristik penting LP: solusi optimal selalu di vertex (titik sudut) daerah feasibel, dan di vertex umumnya banyak variabel bernilai 0 (kendala equality + non-negativity yang aktif). Jadi solusi pecah hanya di sebagian variabel, bukan kombinasi semua produk. Reduced cost untuk B1 (= profit aktual − contribution dari sumber daya yang dipakai, dievaluasi dengan dual) adalah negatif — artinya menambah 1 unit B1 akan menurunkan Z optimal dari (0, 18.75, 0, 30). Semua jam welding yang tersedia (150) habis untuk B2 (2·18.75 = 37.5) dan B4 (3·30 = 90). Sisa welding 22.5 tidak cukup untuk B1/B3 plus B2/B4 secara bersamaan tanpa menurunkan Z. Action manajerial: bicarakan ulang kontrak B1/B3 — apakah profitnya bisa naik agar masuk mix? Atau diskontinuasi formal kalau memang permanen tidak optimal.
❌ Periksa kembali. B1 punya profit 30 (positif), bahkan B1 punya rasio profit/welding tertinggi (30). Tetapi tidak masuk mix optimal — alasannya bukan profit absolut, tetapi opportunity cost di kombinasi semua kendala. Inilah kekuatan LP: ia menyelesaikan kompromi multi-kendala yang tidak intuitif kalau dilihat per produk saja.

✅ Tepat! Strategi bertahap (staged investment) adalah pendekatan optimal di bawah ketidakpastian. Overtime (A) memberi marginal profit Rp 8k/jam dengan komitmen rendah dan reversibel — coba selama 1–3 bulan, validasi asumsi linieritas (apakah profit per unit B2/B4 konstan saat volume naik?), dan kumpulkan data demand growth. Mesin baru (B) capex Rp 800 juta vs marginal benefit Rp 48 juta/tahun → payback 16.7 tahun — tidak feasible kecuali demand future naik 2-3× lipat. Outsource (C) marginal profit Rp 4k/jam (lebih rendah dari overtime A) tetapi tidak ada modal awal. Untuk peak demand musiman atau saat overtime sudah maksimal, outsource bisa menjadi complement. Caveat: shadow price hanya valid dalam rentang sensitivity (lihat res.ineqlin.marginals dan range b yang preserve basis); kalau kapasitas welding ditambah terlalu banyak, kendala lain (CNC, painting, material) bisa menjadi bottleneck baru — re-solve LP.
❌ Periksa kembali. Modernisasi pabrik (B) hanya bagus jika finansial-nya bagus — payback 16.7 tahun > lifespan 8 tahun mesin = NPV negatif. Outsource (C) marginal profit 4k vs overtime (A) yang 8k, jadi (A) lebih dulu menguntungkan. "Tidak berubah" adalah meninggalkan profit di meja: shadow price Rp 16k berarti SETIAP jam welding tambahan worth Rp 16k untuk perusahaan.

Validasi Asumsi LP: kapan model linier "rusak" di skenario nyata? Bagaimana mengantisipasi dan kapan harus pindah ke NLP/MILP?

›

LP mengasumsikan linearitas, divisibility, dan certainty. Di pabrik nyata, asumsi ini sering runtuh: profit per unit turun saat volume tinggi (volume discount pelanggan), biaya overtime naik kuadratik setelah jam tertentu (kelelahan operator), unit produksi harus integer (tidak bisa 18.75 unit B2 — harus 18 atau 19), demand fluktuatif minggu-ke-minggu. Diskusi: di kasus pabrik bracket ini, kapan LP biasa mulai memberikan keputusan menyesatkan? Bagaimana mengetahui itu? Tools mana yang harus digunakan setelah LP "rusak"?

linieritas: profit/unit konstan? divisibility: x harus integer? certainty: parameter c, A, b stokastik?
💬 Tidak ada jawaban tunggal — yang dinilai adalah pemahaman batasan model. Pertimbangan: (1) volume discount → biaya bukan linier; pakai piecewise linear approximation atau NLP. (2) integer requirement → pakai MILP (scipy.optimize.milp atau PuLP/Pyomo). (3) demand fluktuatif → Stochastic LP atau Robust Optimization (worst-case). (4) overtime kuadratik → QP (Quadratic Programming) atau NLP. (5) cek post-optimal: bagaimana solusi berubah kalau parameter berubah ±5%? Sensitivity analysis bawaan LP membantu — sampai batas tertentu. (6) tools selanjutnya di kursus: Pertemuan 10 → NLP, Pertemuan 11 → metaheuristik (GA/PSO) untuk masalah non-konveks.
