Bagian 01

Pipeline Akuisisi Data:
Dari Sensor ke Analisis

Sebelum data time series sampai ke notebook Anda, ia melewati pipeline panjang: sensor fisik → signal conditioning → ADC → bus komunikasi → edge device → storage → analytics. Tiap tahap punya potensi memperkenalkan artefak, latensi, atau loss — engineer time series harus paham seluruh chain ini agar bisa debug bila data tidak sesuai ekspektasi. Pertemuan ini memecah pipeline jadi komponen-komponennya dan menjelaskan trade-off di tiap titik.

Pipeline Akuisisi — Lima Tahap Utama
Sensor → Conditioning → ADC → Edge → Storage/Cloud → Analytics
🔌

Sensor → Signal Conditioning

Sensor menghasilkan tegangan/arus analog yang lemah. Signal conditioning meliputi: amplifikasi (gain), filtering (anti-aliasing low-pass), isolation (galvanic untuk safety), level shifting (bias DC). Tahap ini di-implement di hardware (op-amp circuit, instrument amp). Kualitas conditioning menentukan SNR akhir yang sampai ke ADC.

Vamp = G·Vsensor + Voffset
🔢

ADC: Resolution & Rate

ADC (Analog-to-Digital Converter) mengubah V analog ke nilai digital. Dua parameter kunci: resolusi (bit) — 12-bit = 4096 level, 16-bit = 65536, 24-bit = 16M (DR ~144 dB); sampling rate — fs harus ≥ 2×f_max sinyal (Nyquist). Trade-off: resolusi tinggi + rate tinggi = data besar & mahal. Pilih sesuai aplikasi.

DR(dB) ≈ 6.02·N + 1.76
📡

Edge Device & Bus

Setelah ADC, data digital ditangani edge controller: STM32, ESP32, Raspberry Pi, atau industrial gateway. Bus komunikasi: I2C/SPI (sensor on-board), Modbus RTU/TCP, Ethernet/IP, MQTT untuk cloud. Edge device sering melakukan preprocessing (RMS, FFT) untuk reduce bandwidth — kirim feature, bukan raw data.

BW = fs · bits · channels
☁️

Storage & Cloud

Data mengalir ke storage: edge buffer (RAM/Flash, hari-mingguan), local server (TimescaleDB, InfluxDB), cloud platform (AWS Timestream, Azure IoT). Pilih berdasarkan: latensi yang dapat ditoleransi (real-time vs batch), retention period, query patterns (aggregate vs detail). Tier storage = practical pattern.

latency vs retention
Mengapa Pipeline End-to-End Penting?
Banyak masalah analisis sebenarnya akar masalahnya di jauh sebelum tahap analisis. Contoh: peak FFT tidak akurat ⟶ ternyata gain ADC saturated. Mean drift bertahap ⟶ ternyata thermal drift di amplifier. Spike acak ⟶ ternyata EMI di kabel sinyal yang tidak shielded. Engineer time series yang baik berfikir vertikal across the stack — dari analytics turun ke fisika sensor.
💡 Posisi Modul 3 dalam Alur Mata Kuliah: P1 (memahami sinyal) → P2 (membersihkan dengan filter) → P3 (Anda di sini): bagaimana sinyal sampai ke Anda & bagaimana memvisualisasikannya secara informatif → P4 (preprocessing untuk ML) → P5–P7 (advanced techniques). Tanpa pemahaman akuisisi, Anda hanya jadi konsumen data — tidak bisa diagnose masalah saat data terlihat aneh.
Bagian 02

Sensor & ADC:
Konfigurasi yang Tepat

Pemilihan sensor dan konfigurasi ADC adalah keputusan paling foundational di akuisisi data. Salah pilih di sini, semua tahap downstream akan terdampak — tidak ada filter atau algoritma yang bisa "menyelamatkan" data yang under-sampled atau under-resolved. Bagian ini memetakan jenis sensor, range frekuensi, dan parameter ADC yang sesuai untuk aplikasi mesin industri.

Pemilihan ADC — Range & Resolusi
LSB = Vrange / 2N  ·  SNRquantization = 6.02·N + 1.76 dB
📍

Akselerometer Vibration

Tipe utama: IEPE/ICP (industri standar, output 100 mV/g, range 0.5 Hz–10 kHz), MEMS (murah, integrated, range DC–1 kHz). Untuk vibration motor: pilih bandwidth ≥ 5× rotational speed dan ≥ 2× ekspektasi bearing fault frequency. Mounting kritis: stud (terbaik), magnetic (cukup), adhesive (buruk untuk frekuensi tinggi).

100 mV/g (IEPE)
🌡️

Temperature Sensor

Tipe: thermocouple K/J/T (range −200°C–1300°C, akurasi ±0.5°C), RTD Pt100 (akurasi ±0.1°C, lebih lambat), thermistor NTC (cepat, range terbatas). Sampling rate untuk monitoring proses: 1–10 Hz cukup (dinamika thermal lambat). Beda dengan vibration yang butuh kHz!

Pt100: R(t) = 100·(1+α·t)
⚡

ADC Resolution Trade-off

12-bit: 4096 level, SNR ~74 dB — cukup untuk monitoring umum (temperatur, pressure). 16-bit: 65k level, SNR ~98 dB — standard untuk vibration. 24-bit Sigma-Delta: 16M level, SNR ~144 dB — high-end untuk audio & precision vibration. Pilih sesuai dynamic range yang dibutuhkan.

DR16-bit ≈ 96 dB
⚙️

Sampling Rate Selection

Aturan praktis untuk fs: (1) oversampling 5–10× dari frekuensi sinyal tertinggi yang ingin dianalisis (margin untuk anti-aliasing filter). (2) Untuk vibration motor 1500 RPM: f1×=25 Hz, harmonik sampai 40×=1 kHz, bearing fault sampai 5 kHz → fs minimum = 25–50 kHz. (3) Pertimbangkan storage budget — fs 10× lebih tinggi = 10× lebih banyak data.

fsrec = 5–10·fmax

⚠️

Kesalahan Konfigurasi ADC yang Umum

(1) Range terlalu lebar — input 100 mV pada ADC ±10V hanya pakai 1% range → kehilangan 7 bit efektif. Pakai gain amplifier sebelum ADC. (2) Range terlalu sempit — clipping/saturation, sinyal terpotong → distorsi parah di FFT. Pastikan headroom ≥ 50%. (3) Lupa anti-aliasing filter analog — frekuensi tinggi yang tidak sengaja ditangkap akan ter-fold ke pita rendah, indistinguishable. (4) Crosstalk antar channel di multiplexed ADC → channel-channel saling mempengaruhi — solusi: lower scan rate atau pakai simultaneous-sampling ADC.

Bagian 03

Storage & Streaming
Time Series

Setelah data digital tersedia di edge, pertanyaan berikutnya: kemana ia disimpan, dan dengan latency seperti apa diaksesnya?. Vibration sensor 10 kHz menghasilkan 70 MB/jam per channel — 1.7 GB/hari. 8 motor = 13.6 GB/hari. Tidak realistis menyimpan semua raw data berbulan-bulan. Bagian ini membahas tiered storage architecture dan tooling time series database modern.

Bandwidth & Storage Calculation
BW (B/s) = fs · (bits/8) · channels  ·  Storage = BW · duration
💾

Edge Buffer (Hari)

Raw data disimpan di edge device (NVMe SSD industrial, 256–1TB) sebagai circular buffer — hari ke-15 menimpa hari ke-1. Format: binary (terbaca cepat) atau Parquet (compressed columnar). Edge buffer melayani query investigasi (mis. "tunjukkan vibration motor-3 1 jam sebelum trip").

FIFO: hari N → hari N+15
🗄️

Time Series Database

TSDB dirancang khusus untuk data berurutan-waktu: InfluxDB, TimescaleDB (PostgreSQL extension), Prometheus, QuestDB. Mendukung downsampling otomatis (raw → 1-min → 1-hour aggregates), retention policy per tier, dan query syntax time-aware (range, window, fill).

SELECT mean(rms) FROM ... GROUP BY time(1h)
📤

Streaming Protocol

Untuk real-time monitoring: MQTT (lightweight pub/sub, QoS 0-2, ideal IoT), Kafka (high throughput, replay capable, enterprise), OPC UA (industri standard, complex info modeling). MQTT cukup untuk 10s of motors; Kafka untuk 1000s. Pilih berdasarkan scale & durability requirements.

QoS: 0=fire&forget, 2=exactly once
📊

Aggregation Strategy

Pattern populer untuk vibration: edge hitung RMS, peak, CF per 1 detik dari raw 10 kHz → kirim ke cloud (1 nilai/detik = 86k nilai/hari, sangat ringan). Raw data tetap di edge untuk forensic analysis bila ada anomali. Cloud-side aggregate lagi: 1-min, 1-hour, 1-day untuk dashboard. Hemat bandwidth, hemat storage, tetap actionable.

edge: 10kHz → cloud: 1Hz

Format Data Time Series:
CSV: human-readable tapi besar & lambat parse — hanya untuk export. Parquet: columnar, compressed, fast query — standard untuk batch analytics. HDF5: hierarchical, support metadata kaya — populer di scientific. TDMS (NI proprietary): standard di industri vibration. InfluxDB line protocol: native untuk TSDB. Pilih format sesuai workflow downstream — jangan pakai CSV untuk data > 10k baris.
💡 Pattern "Hot/Warm/Cold": Tiered storage adalah cost-saving best practice — data terbaru & sering diakses di storage cepat (SSD, in-memory cache), data lama yang jarang di tier murah (object storage, cold cloud). Auto-tiering policy di TSDB modern memindahkan data sesuai umur. Tanpa policy ini, biaya storage akan meledak setelah beberapa bulan operasi 24/7.
Bagian 04

Visualisasi
Time Series Industri

Plot adalah cara tercepat untuk pahami data — tapi plot yang buruk memperburuk pemahaman. Bagian ini membahas best practice visualisasi time series industri: kapan pakai line plot vs spectrogram vs heatmap, bagaimana handle data multi-channel + multi-rate, dan tooling modern (matplotlib, plotly, Grafana). Ingat: good visualization = good decisions.

Empat Jenis Plot Time Series Utama
Line · Spectrogram · Heatmap · Statistical (boxplot, violin)
📈

Line Plot — Default

Line plot adalah visualisasi default untuk time series — y vs t. Best practice: (1) sampling display ≤ 10k titik (downsample dengan max/min preservation untuk plot panjang), (2) title yang informatif (apa, kapan, dari mana), (3) y-axis dengan unit eksplisit, (4) grid alpha rendah (0.3) untuk readability, (5) jika multi-line, pakai colormap yang accessible (viridis, bukan rainbow).

plt.plot(t, y); plt.grid()
🌈

Spectrogram

Spectrogram = FFT bergulir di window — menampilkan bagaimana frekuensi sinyal berubah seiring waktu. Sumbu: t (horizontal) × f (vertikal), warna = amplitudo. Sangat berguna untuk: motor startup/shutdown analysis (frekuensi naik-turun), bearing degradation (peak frekuensi muncul perlahan), gear mesh modulation. Pakai scipy.signal.spectrogram.

spectrogram(x, fs)
🟦

Heatmap (Multi-Sensor)

Untuk pabrik dengan 100+ motor, line plot semua sensor jadi clutter. Heatmap dengan baris=sensor, kolom=waktu, warna=metric (mis. RMS) jauh lebih readable. Pattern abnormal langsung terlihat sebagai band warna yang mencolok. Pair dengan annotation otomatis (label sensor di luar threshold). Tooling: matplotlib imshow, seaborn heatmap, plotly.

heatmap(rms_matrix)
📦

Boxplot & Violin

Statistical plots untuk membandingkan distribusi sinyal antar shift, antar bulan, antar mesin. Boxplot: median, quartile, outlier. Violin: kombinasi boxplot + density estimate (lebih informatif tapi lebih kompleks). Sangat efektif untuk identifikasi shift drift (mean shift, variance change) yang tidak terlihat di line plot harian.

matplotlib: gold standard untuk publikasi, paper, report — kontrol presisi, statis. plotly: interaktif, zoom/pan, hover info — bagus untuk Jupyter exploration. ⚠️

Anti-Pattern Visualisasi yang Harus Dihindari

(1) Plot ribuan titik tanpa downsampling — browser hang, file PDF jadi puluhan MB. Pakai max/min downsampling untuk visual; hitung statistik untuk numerical. (2) Y-axis auto-scaling tanpa baseline — nilai 100±0.1 terlihat seperti chaos karena auto-scale ke 0.2 range. Set ylim manual atau gunakan log scale. (3) Rainbow colormap untuk continuous data — tidak perceptually uniform, bias terhadap kuning. Pakai viridis, plasma, atau cividis. (4) Plot tanpa unit di axis — "RMS: 1.5" — 1.5 apa? mm/s? g? V? Selalu unit eksplisit.

Bagian 05

Enam Analogi Akuisisi
untuk Memori Visual

Pipeline akuisisi data dan visualisasi terasa abstrak hingga Anda menemukan analogi yang relevan. Berikut enam analogi dari kehidupan sehari-hari yang membuat sensor chain, tiered storage, dan data visualization menjadi intuitif. Setiap kali Anda lupa, ingat analogi ini.

🎤

Mikrofon Studio Rekaman

Setup mikrofon studio = sensor chain industrial: mic → preamp (gain) → audio interface (ADC) → DAW software (storage). Tiap tahap bisa hancurkan kualitas: mic murah → SNR rendah; gain salah → clipping; bit rate kurang → quantization noise. Persis seperti sensor industri: akselerometer murah, op-amp salah, ADC under-resolved.

mic → preamp → ADC → DAW
📷

Kamera DSLR vs HP

Sensor full-frame DSLR vs sensor kecil HP: dynamic range berbeda. DSLR 14-bit RAW = 16384 level cahaya, detail di bayangan & highlight terjaga. HP 8-bit JPEG = 256 level, easily clipped. Persis seperti pilihan ADC: 24-bit untuk vibration high-end (DR 144 dB), 12-bit untuk monitoring kasar.

DR = 6.02·N + 1.76 dB
📡

Streaming Netflix vs Download

Netflix streaming: bandwidth pendek tapi continuous, buffer kecil (latency rendah, langsung tonton). Download: bandwidth besar, simpan sekali pakai berkali-kali (offline access). Persis seperti edge buffer (streaming) vs cloud archive (download). Pilih sesuai pola akses.

streaming vs storage trade-off
🗺️

Google Maps Zoom Levels

Google Maps tiered tiles: zoom dunia = 1 tile blur, zoom kota = ratusan tiles, zoom street = jutaan tiles. Saat zoom out, server tidak streaming detail rumah — kasih agregasi. Persis seperti time series tiered storage: hourly aggregate untuk dashboard daerah luas, raw 10 kHz untuk forensic zoom in.

level-of-detail by zoom
📺

Dashboard Pesawat / Mobil

Dashboard mobil = visualisasi multi-sensor real-time. Speedometer (1 Hz update cukup), tachometer (10 Hz responsif), warning light (binary status), engine temp (1 Hz, stabil). Setiap sensor pakai visualisasi berbeda sesuai karakter datanya. Persis seperti Grafana dashboard: gauge untuk current value, trend untuk history, alert untuk threshold.

visualization fits data character
🌡️

Termometer vs Thermal Imaging

Termometer klinis: 1 angka per pengukuran, akurat tapi single-point. Thermal imaging camera: matriks 480×640 piksel temperatur, lihat distribusi spatial. Persis seperti single sensor (line plot) vs sensor array (heatmap). Beda kasus, beda visualisasi optimal.

scalar vs spatial
Mengapa Banyak Analogi?
Data acquisition adalah konsep yang muncul di banyak domain: audio engineering, photography, medical imaging, networking, scientific instrumentation. Engineer time series yang baik dapat melihat pipeline-nya di mana-mana dan menggunakan intuisi dari domain lain untuk debug masalah. Banyak praktisi vibration sebenarnya datang dari background audio — mereka langsung paham SNR, anti-aliasing, dynamic range.
Bagian 06

Visualisasi Pipeline
Akuisisi Data

Empat animasi kanvas berikut memvisualisasikan tahap-tahap pipeline akuisisi: (1) signal flow dari sensor analog hingga ADC digital, (2) tiered storage dengan data mengalir dari edge ke cloud, (3) spectrogram live dari motor running dengan frekuensi yang berubah, (4) multi-sensor heatmap dengan deteksi anomali per-mesin.

Animasi 1 — Deret Mentah: Tren + Musiman + Noise Interaktif
📊 Cara Membaca Animasi: Ungu = deret asli, kombinasi dari tren (biru) + musiman (oranye) + noise acak. Tren dan musiman bisa Anda atur slidernya.
Coba: (1) Set Tren = 0 dan Musiman = 1 → hanya komponen periodik murni yang terlihat. (2) Set Tren = 2 dan Musiman = 0 → tren linear murni. (3) Set keduanya tinggi → deret kompleks khas data industri — sulit dibaca mentah, butuh dekomposisi.
Animasi 2 — Log Transform: Menstabilkan Varians yang Tumbuh
📊 Cara Membaca: Kiri = deret asli dengan varians yang tumbuh (heteroskedastik) — osilasi kecil di awal, besar di akhir. Kanan = hasil log(y): osilasi menjadi hampir seragam amplitudonya.
Coba: Geser faktor pertumbuhan ke 3.0 — deret asli jadi "corong" ekstrem. Log transform memampatkannya jadi strip horizontal. Inilah mengapa data energi, produksi, dan finansial hampir selalu di-log-kan sebelum dimodelkan.
Animasi 3 — Differencing: Dari Deret Bertren ke Stasioner
📊 Cara Membaca: Atas = deret asli dengan tren linear naik. Bawah = setelah differencing ∇dxt. Orde d = 0 → tidak ada perubahan. d = 1 → tren linear hilang. d = 2 → tren kuadratik hilang.
Coba: d = 0 → deret naik-naik-naik (non-stasioner). d = 1 → fluktuasi di sekitar nol (stasioner!). Perhatikan: tiap orde differencing mengurangi 1 observasi — overdifferencing memperbesar noise, jadi jangan lebihi d yang diperlukan.
Animasi 4 — Dekomposisi Aditif: Memecah Deret ke T + S + R
📊 Cara Membaca: Empat panel dari atas ke bawah: Observasi (ungu) = data asli; Trend (cyan) = hasil moving average; Seasonal (amber) = pola periodik sisa; Residual (hijau) = noise acak yang tersisa.
Coba: Window 4 → tren terlalu bergejolak (under-smoothing). Window 12 → tren halus, musiman dan residual terpisah bersih. Window 30 → over-smoothing, pola musiman bocor ke residual. Memilih window = memilih periode musiman yang Anda percaya.
Bagian 07

Aplikasi Akuisisi
di Industri

Mari lihat empat sistem akuisisi data nyata dari industri Indonesia. Setiap kasus mengilustrasikan kombinasi sensor + ADC + storage + visualisasi yang berbeda — menunjukkan bahwa tidak ada satu arsitektur yang cocok untuk semua aplikasi.

🏭

Vibration Monitoring Pabrik Semen

20+ motor mill grinding, 1-15 MW. Setup: IEPE accelerometer per motor, 16-bit ADC fs=25 kHz, edge gateway hitung RMS+CF per detik, kirim via MQTT ke InfluxDB. Dashboard Grafana: line plot RMS 24h + heatmap 30-day overview semua motor. Alert tier: notice (RMS>2.8), warning (>4.5), critical (>7.1) sesuai ISO 10816 kelas II.

25 kHz · 16-bit · MQTT
🌾

SCADA Kelapa Sawit

Pabrik kelapa sawit: ratusan sensor temperatur (RTD/TC), pressure (4-20mA), level (ultrasonic). Setup: PLC Allen-Bradley dengan analog input modules, fs=1 Hz, scan time 100ms. Komunikasi Modbus TCP → SCADA Wonderware. Dashboard: P&ID dengan live values per tag. Storage: SQL Server historian, retention 5 tahun untuk compliance.

1 Hz · 16-bit · Modbus
⚡

Power Quality Gardu Distribusi

Gardu distribusi 20 kV: 3-phase voltage + current. Setup: CT/PT trafo step-down → 20 kHz simultaneous-sampling 16-bit ADC → real-time FFT untuk THD, harmonik (sampai orde 50). IEC 61850 protocol untuk integrasi dengan SCADA grid. Output: power quality metrics per 10 menit (sesuai standar EN 50160).

20 kHz · IEC 61850
🏥

Patient Monitor ICU

ICU patient monitor: ECG (500 Hz, 12-bit), SpO2 (1 Hz), NIBP (event-based), temp (0.1 Hz). Multi-rate sampling di satu device — challenge: sinkronisasi timestamp lintas sensor. Storage: HL7 FHIR di hospital information system. Visualization: dashboard waveform 5 detik live + trend metrics 1 jam. Alert berbasis algoritma per metric.

multi-rate sync
⚠️

Lessons Learned dari Industri

(1) Lupa cable shielding di pabrik dengan banyak inverter VSD — sinyal sensor terkontaminasi EMI parah. Pakai shielded twisted pair, ground di satu sisi saja. (2) Edge gateway timeout settings terlalu agresif — jaringan flapping → data loss kontinyu. Pakai store-and-forward dengan buffer minimal 1 jam. (3) Tidak ada timestamp synchronization antar gateway → cross-sensor analysis salah. Pakai NTP atau PTP (IEEE 1588). (4) Dashboard query terlalu berat di TSDB → loading lama, user frustrated. Pakai pre-computed aggregates & downsampling.

Cell 1 — Simulasi Multi-Sensor Akuisisi
demo
import numpy as np
import pandas as pd

# Simulasi 4 motor pabrik, 1 jam data, vibration RMS per detik
np.random.seed(42)
N_motors = 4
duration_sec = 3600   # 1 jam
fs_metric = 1         # 1 Hz untuk RMS aggregat (bukan raw)
N = duration_sec * fs_metric

# Tiap motor punya baseline berbeda + noise + occasional event
data = {}
timestamps = pd.date_range('2026-01-01 08:00:00', periods=N, freq='1s')
for i in range(N_motors):
    baseline = 1.5 + 0.3 * i             # baseline 1.5, 1.8, 2.1, 2.4 mm/s
    noise = np.random.normal(0, 0.15, N)
    # Motor 2 dapat anomali sekitar menit 30-35
    event = np.zeros(N)
    if i == 1:
        event[1800:2100] = np.linspace(0, 1.5, 300)  # gradual rise
    rms = baseline + noise + event
    data[f'motor_{i+1}_rms'] = rms

Cell 3 — Spectrogram: Time-Frequency Visualization

from scipy.signal import spectrogram

# Generate vibration signal dengan frekuensi yang berubah (motor startup)
np.random.seed(7)
fs = 5000          # 5 kHz sampling
T_dur = 4.0        # 4 detik startup
N2 = int(fs * T_dur)
t2 = np.linspace(0, T_dur, N2)

# Frekuensi sweep dari 5 Hz ke 25 Hz selama startup
freq_inst = 5 + 5 * t2
phase = 2 * np.pi * np.cumsum(freq_inst) / fs
sig_startup = np.sin(phase) + 0.1 * np.random.randn(N2)
mask = t2 > 2.0
sig_startup[mask] += 0.4 * np.sin(2 * phase[mask])
sig_startup[mask] += 0.2 * np.sin(3 * phase[mask])

# Compute spectrogram
f, t_spec, Sxx = spectrogram(sig_startup, fs=fs, nperseg=512, noverlap=256)

# Plot
fig, axes = plt.subplots(2, 1, figsize=(11, 7), sharex=True)
axes[0].plot(t2, sig_startup, lw=0.5, color='#0ea5e9')
axes[0].set_ylabel('Amplitude (m/s²)')
axes[0].set_title('Time domain — Motor startup signal')
axes[0].grid(alpha=0.3)

im = axes[1].pcolormesh(t_spec, f, 10*np.log10(Sxx + 1e-10),
                       shading='gouraud', cmap='viridis')
axes[1].set_ylabel('Frekuensi (Hz)')
axes[1].set_xlabel('Waktu (detik)')
axes[1].set_ylim(0, 100)
axes[1].set_title('Spectrogram — Motor startup (frekuensi naik + harmonik muncul)')
plt.colorbar(im, ax=axes[1], label='Power (dB)')
plt.tight_layout(); plt.show()

# Verifikasi: temukan frekuensi dominan di t=3.5s
idx_3p5 = np.argmin(np.abs(t_spec - 3.5))
spec_at_3p5 = Sxx[:, idx_3p5]
peak_freq = f[np.argmax(spec_at_3p5)]
print(f"Frekuensi dominan di t=3.5s: {peak_freq:.1f} Hz (expected ~22.5 Hz)")
⤷ Output: 2-panel plot. Time domain menunjukkan amplitudo & frekuensi yang berubah; spectrogram dengan jelas memperlihatkan ramp frekuensi 5→25 Hz + harmonik muncul di t=2s. Pattern kunci untuk diagnosis startup motor.

🎯 Scalable ke 100+ motor tahun 2

Kebutuhan output: (1) real-time dashboard untuk control room (refresh detik), (2) historical analytics untuk reliability engineer (trend bulanan, root cause analysis), (3) compliance audit trail 5 tahun (regulasi industri tekstil). Bagas memiliki 6 minggu untuk: pilih sensor, design akuisisi pipeline, pilih TSDB & visualization tool, deploy MVP. Bagas minta saran tim mahasiswa magang untuk memvalidasi keputusan arsitektural utama sebelum procurement dimulai.

01

Arsitektur akuisisi (sensor + ADC + edge gateway) yang Anda rekomendasikan untuk Bagas?

›

Sensor & ADC selection adalah foundational decisions — sulit di-undo setelah procurement. Rancang spesifikasi: (1) tipe akselerometer untuk 30 motor (mempertimbangkan range frekuensi yang berbeda untuk RPM 720 vs 2900), (2) sampling rate & ADC resolusi yang sesuai untuk vibration vs slow process (suhu/pressure), (3) edge gateway architecture (centralized vs per-line vs per-motor). Diskusikan trade-off cost vs reliability vs scalability ke 100+ motor tahun 2.

IEPE 100mV/g fs vibration: 25–50 kHz fs process: 1–10 Hz
QUICK CHECK — Arsitektur akuisisi yang paling tepat?
Single-rate ADC fs=50 kHz untuk semua sensor (vibration + suhu + pressure) — uniformity
Multi-rate: IEPE+24-bit ADC 25 kHz untuk vibration, RTD/TC+16-bit ADC 1-10 Hz untuk process, edge gateway per-line (cluster 10 motor) untuk scalability
Cloud-only — kirim raw data dari semua sensor langsung ke cloud tanpa edge processing
MEMS accelerometer murah saja (range terbatas) untuk hemat budget
✅ Tepat! Multi-rate adalah arsitektur industri standar: (1) vibration butuh fs tinggi (kHz) untuk capture bearing fault frequencies hingga 5-10 kHz; (2) process sensor bisa fs rendah (1-10 Hz) — dinamika thermal/pressure lambat, fs tinggi hanya boros storage tanpa benefit. 24-bit ADC untuk vibration high-end memberi DR ~144 dB — penting untuk capture amplitude range luas dari noise lantai sampai impulsive event. Edge gateway per-line (cluster 10 motor): scalable horizontal (tambah cluster untuk 100 motor), dan store-and-forward melindungi dari network outage. Kirim hanya feature (RMS, peak, FFT bins) ke cloud — bukan raw data.
❌ Single-rate 50 kHz untuk semua sensor = boros storage 100× untuk process data (suhu yang dinamikanya 1 Hz disampling 50 kHz). Cloud-only tanpa edge = bandwidth requirement absurd (30 motor × 25 kHz × 16-bit = 12 MB/s); juga tidak resilient terhadap network outage. MEMS murah punya range terbatas (DC-1 kHz) — tidak bisa capture bearing fault 1-10 kHz, salah pilih sensor untuk aplikasi.

02

Storage architecture (TSDB + tiered storage) seperti apa yang Anda rancang untuk Bagas?

›

Bagas perlu balance antara: (a) real-time access untuk dashboard (millisecond latency), (b) historical analytics 1 tahun (bulanan reliability review), (c) compliance archive 5 tahun (audit). Total data volume bisa terabytes — naive single-tier storage akan boros & lambat. Rancang strategy: pilih TSDB (InfluxDB/TimescaleDB/Prometheus), tier hot/warm/cold, downsampling policy, retention per tier. Justifikasi pilihan dengan estimasi cost.

💬 Pertimbangkan: 30 motor × edge feature 1 Hz (RMS, peak, CF, kurtosis) = 4 nilai/detik/motor = ~10 juta nilai/hari. 70 process sensor × 1 Hz = 6 juta nilai/hari. Total ~16M points/day. Hot tier (1 bulan, raw): butuh ~500M points di SSD. Warm tier (1 tahun, downsampled 1-min): ~9M points. Cold tier (5 tahun, downsampled 1-hour): ~14M points. Tier policy menghemat >10× storage cost vs simpan raw selama 5 tahun.
QUICK CHECK — Storage architecture yang paling tepat?
Pakai PostgreSQL biasa untuk semua data — sederhana
InfluxDB Cloud (TSDB) dengan retention policy: hot 30 hari (raw), warm 1 tahun (1-min downsampled), cold 5 tahun (1-hour) di S3 Parquet untuk audit
Spreadsheet Excel multi-sheet untuk semua history
Flat files CSV per hari di network share, no database
✅ Tepat! TSDB dirancang khusus untuk time series workload: (1) InfluxDB punya native time-aware queries, downsampling otomatis, retention policy builtin, dan integrasi out-of-the-box dengan Grafana. (2) Tiered retention: hot 30 hari raw cukup untuk recent forensic; warm 1 tahun 1-min agregat cukup untuk monthly reliability review; cold 5 tahun 1-hour di S3 Parquet sangat hemat untuk audit compliance (storage cost ~$0.023/GB/month vs SSD $50+/GB). (3) Estimasi total cost: ~30 GB hot + 5 GB warm + 2 GB cold ≈ < $50/bulan cloud cost untuk 30 motor — sangat ekonomis.
❌ PostgreSQL biasa tidak optimal untuk time series — tidak ada native downsampling/retention, query window function lambat. TimescaleDB (extension) bisa, tapi Influx lebih sederhana untuk kasus ini. Excel obviously tidak scalable untuk 16M points/hari. CSV flat files = no indexing, query trend bulanan butuh scan TB data — lama & lambat. Tanpa retention policy, biaya storage akan eksponensial di tahun 2-3.

03

Dashboard visualization strategy: bagaimana mendesain "control room wall" vs "reliability engineer dashboard" vs "executive summary"?

›

Tiga audience yang sangat berbeda butuh visualisasi berbeda. Control room operator butuh status realtime (gauge, trend pendek, alerts) — refresh detik. Reliability engineer butuh deep-dive (spectrogram, multi-sensor correlation, historical comparison) — interactive exploration. Executive butuh high-level KPI (uptime %, OEE, cost saved) — bulanan summary. Rancang 3 dashboard berbeda dengan plot type yang sesuai untuk tiap audience & tooling yang tepat.

Control room: Grafana Engineer: Jupyter / Streamlit Executive: Power BI / Tableau
💬 Tidak ada solusi tunggal — yang dinilai adalah pemahaman audience-first design: visualisasi mengikuti pertanyaan yang dijawab pengguna, bukan sebaliknya. Operator bertanya "ada masalah sekarang?" → status + alerts. Engineer bertanya "kenapa motor ini bermasalah?" → drill-down + correlation. Executive bertanya "berapa OEE bulan ini vs target?" → KPI + trend. Plot dan tooling mengikuti pertanyaan, bukan dipilih dulu.
